Section 1 — Why Functions?
1

Why Functions? 0–15 min

Concept lecture · Two motivations: reuse and testability

🗣 Say — the reuse argument
“Without functions, every business rule is written out in full every time it is needed. If the approval threshold changes from $1,000 to $2,000, you must find and update every copy. Miss one and you have a bug. With a function, you update it in one place and every caller gets the new behaviour.”
🗣 Say — the testability argument
“You cannot write a pytest test for an anonymous block of code. Naming the logic is what makes it testable. Every function in business_rules.py will have a corresponding test. If you cannot test it, you cannot trust it.”
✏ Two reasons to write a function — write on board
1. You need to use this logic more than once.    2. You need to be able to test this logic independently.
Section 2 — Defining, Calling, and Returning
2

Defining, Calling, and Returning 15–35 min

Live demo · Business logic examples only

🗣 Say — before demo
“A function definition is the blueprint. A function call is the execution. A return value is the output. These three concepts are the entire function model. Everything else is detail.”
Build three business rule functions together
def calculate_total(price: float, quantity: int) -> float:
    """Return total cost including 7% tax."""
    subtotal = price * quantity
    tax = subtotal * 0.07
    return subtotal + tax

def requires_review(amount: float) -> bool:
    """Return True if amount exceeds the approval limit."""
    return amount > 1000

def get_approval_tier(amount: float) -> str:
    """Return the approval routing tier."""
    if amount <= 500:    return 'auto'
    elif amount <= 2000: return 'manager'
    else:                return 'director'
🗣 Say — after writing the functions
“Notice that none of these functions contain a print(). They return values. main.py calls them and prints the results. This separation — logic in functions, display in main — is how professional code is structured and how tests work.”
Section 3 — Scope
3

Scope 35–55 min

Short but critical · Show the NameError live

🗣 Say — what scope means
“A variable defined inside a function is local — it does not exist outside that function. This is a feature, not a limitation. It means functions cannot accidentally overwrite each other's data. Always pass data in via parameters and get it back via return.”
⚠
Show the NameError live
Define a function that creates a local variable. Then try to print that variable outside the function. Get the NameError. Say: “This error message is Python telling you exactly what is wrong. Read it. Do not ask AI to fix a NameError before you understand what scope means.”
Section 4 — Modules and Required Project Structure
4

Modules and Required Project Structure 55–70 min

Live demo · Show the import and test file

🗣 Say — what a module is
“A module is any .py file you import from. When business_rules.py contains your functions, main.py can use them with from business_rules import calculate_total. Tests do the same thing. This is how every professional Python project is organised.”
✏ Required Module 5 project structure — write on board
module05_functions/ — README.md · main.py · business_rules.py · requirements.txt · .gitignore · tests/__init__.py · tests/test_business_rules.py
tests/test_business_rules.py — show the import and test pattern
from business_rules import calculate_total, requires_review

def test_calculate_total():
    assert calculate_total(100.00, 2) == 214.00

def test_requires_review_true():
    assert requires_review(1500) is True

def test_requires_review_boundary():
    assert requires_review(1000) is False  # exactly at limit
💡 The boundary test — make this explicit
Ask: “Why test with exactly 1000?” Look for: because that is the edge case where the rule changes. Off-by-one errors in business rules have real consequences. Boundary tests are the ones that catch the bugs that matter.
