Section 1 — Developer Mindset and Course Identity
1

Developer Mindset and Course Identity 0–15 min

Lecture + two discussion prompts · No setup required yet · No slides needed

🗣 Say — opening
“Welcome to ISM3232. This course is not a programming course and not a tools course — it is a business application development course. The difference matters. A programming course teaches you Python syntax. This course teaches you to build things that work, on a machine configured correctly, submitted cleanly through Git, with tested code. The workflow is the course.”
🗣 Say — follow-up
“Everything we do in the first four weeks — the terminal, the virtual environment, the folder structure, Git — is not preamble before the real content. It is the content. The midterm in Week 9 tests all of it.”
📋 Write on board — leave up for the rest of class
THIS COURSE IS: a professional practice course · a course where workflow is graded alongside code · a course that tests the terminal, venv, ruff, pytest, and Git on the midterm

THIS COURSE IS NOT: a course where you can skip the terminal and pass · a course where AI can substitute for understanding · a course where "final" is an acceptable commit message
💬 Discussion prompt 1
Ask: “Open your computer right now and find where you saved the last file you worked on. How long did it take? Could you do it from a terminal without clicking anything?”

Wait for 3–4 responses. The goal is not to embarrass anyone — it is to surface that most students have never thought consciously about where files live. This opens the paths section naturally.
💬 Discussion prompt 2
Ask: “When a developer says a script ‘runs,’ what exactly is happening? Who is running it, and from where?”

Look for the distinction between double-clicking an application and executing a file from a shell. Most students have never made this distinction. That gap — between the GUI and the command line — is what this week closes.
⚠
Instructor note — set expectations explicitly
Be direct: students who treat Week 1 as orientation and do not complete their setup will fall behind immediately in Week 2. Live coding sessions start in the terminal on Day 1 of Week 2. No working setup = cannot follow the demo. Say this out loud.
Section 2 — Files, Folders, Paths, and Directory Trees
2

Files, Folders, Paths, and Directory Trees 15–35 min

Conceptual lecture · Draw the tree diagram on the board · Students copy it into notes

🗣 Say
“Every file on your computer lives at a specific address. That address is called a path. A path is just a sequence of folder names that tells the computer how to navigate from the top of the disk to your file. Understanding paths is the single most useful thing you will learn in Week 1 of this course.”
✏ Draw this on the board by hand — students copy it
Draw slowly, annotating each level as you go. Do not skip this or use a slide — the act of drawing it while talking is what makes the mental model stick.
Directory tree — draw and annotate
/                           # root of the filesystem (Mac/Linux)
└── Users/
    └── yourname/           # home directory  ~
        ├── Desktop/
        ├── Documents/
        └── ism3232/            # ← course folder lives here
            ├── module01_setup/
            ├── module02_zsh/
            ├── module03_git_github/
            ├── module04_programming/
            ├── module05_functions/
            ├── module06_oop/
            ├── module07_final_project/
            ├── data/
            └── screenshots/

Annotate as you draw, saying out loud:

Root /

“This is the top of everything on Mac and Linux. The entire disk lives under this single slash. There is nothing above it.”

Home directory ~

“This is your home directory. The tilde character ~ is a shortcut for this full path — /Users/yourname. Every time you open a new terminal, you start here. When you get lost, cd ~ brings you back.”

Module folders

“This week, everything you submit lives inside module01_setup/. Next week, inside module02_zsh/. The folder structure maps directly to the assignment structure. This is not accidental.”

🗣 Explain absolute vs relative paths
“An absolute path is the full address from the root. It always works, regardless of where you are. An example:” 
/Users/yourname/ism3232/module01_setup/hello_ism3232.py
🗣 Continue
“A relative path is the address from wherever you are right now. Shorter, but only works if you are in the right place. If you are already inside ism3232/, the same file is just:” 
module01_setup/hello_ism3232.py
📋 Write the symbols table on board — explain each row
/ — root of the filesystem · ~ — your home directory · . — current directory (where you are right now) · .. — one directory up · .py / .md — file extensions, tell the OS what type the file is
💡 Make it concrete
Ask students to find their Downloads folder in Finder or Explorer. Then ask: “What is the full path to that folder?” Most will not know. Then show it in the terminal:

echo ~/Downloads

The gap between clicking and knowing the actual address is the lesson. This is what pwd solves.
Section 3 — VS Code, zsh, and Stack Verification
3

VS Code, zsh, and Stack Verification 35–55 min

Live demo + student follow-along · Hard checkpoint before moving on · Walk the room

⚠
Critical — Windows students
WSL installation failures are the most common Week 1 blocker. Watch the room during Step 2. Have the lab machine fallback address ready. Any student who cannot complete WSL by end of class should be directed to the setup office hour before Week 2 starts.
🗣 Say before starting
“We are going to install and verify every tool together. Do not skip steps. Do not move ahead. If something does not work for you, raise your hand immediately — do not silently move on, because the next step depends on this one.”
1
VS Code — install and Python extension
Download from code.visualstudio.com if not already installed. Extensions panel → search “Python” → install the Microsoft extension. Verify it shows as installed.
2
Windows only — WSL + Ubuntu + zsh
Open PowerShell as Administrator. Run wsl --install. Restart when prompted. Open Ubuntu app, complete UNIX user setup, then run:
sudo apt update && sudo apt install zsh -y
chsh -s $(which zsh)
Restart the terminal. Watch the room carefully during this step.
3
Mac/Linux — verify zsh is the default shell
Run echo $SHELL — should return /bin/zsh. If not: chsh -s /bin/zsh and restart the terminal.
4
Set VS Code integrated terminal to zsh
⌘, (Mac) or Ctrl+, (Windows) → search “terminal default profile” → set to zsh (Mac/Linux) or WSL (Windows). Close and reopen the terminal panel. Prompt should show % or zsh theme.
5
Run the six verification commands — everyone must pass
Walk the room. Do not proceed until every student sees expected output for all six.
Verification commands — students run, instructor walks the room
echo $SHELL          # Expected: /bin/zsh
zsh --version        # Expected: zsh 5.x or higher
python3 --version    # Expected: Python 3.10 or higher
git --version        # Expected: git version 2.x
pwd                  # Expected: /Users/yourname  (home directory)
ls                   # Expected: Desktop  Documents  Downloads  etc.
📸
Screenshot moment — tell students: “Take a screenshot of this output right now. This is Assignment 1 deliverable #3. The screenshot must show all six commands and their output in the zsh terminal.”
✓
Hard checkpoint — do not move on until this is done
Every student must show all six commands returning expected output before you move to Section 4. Pair stuck students with a neighbor or move them to a lab machine. Do not let setup problems prevent participation in the live coding sections.

4

Building the Course Folder Structure 55–65 min

10 minutes · Students follow along in their own terminal

🗣 Say
“We are creating the folder structure for the entire course right now. You will use these folders every week. The naming conventions — lowercase, underscores, module numbers — are not arbitrary. They are professional Python project conventions. If your file is in the wrong folder, it is in the wrong place.”
Type together — students follow along in their own terminal
cd ~
mkdir ism3232
cd ism3232
mkdir module01_setup module02_zsh module03_git_github       module04_programming module05_functions       module06_oop module07_final_project       data screenshots
ls
code .
cd module01_setup
💡 Explain code .
“code . opens VS Code with the current folder as the workspace. The dot means ‘here.’ This is how developers open projects — they navigate to the folder first, then open the editor. It is different from double-clicking a file. The workspace, not the file, is the unit of work. Return to this concept in Week 5 when we talk about module imports.”
Section 5 — First Python Script
5

Writing and Running hello_ism3232.py 65–75 min

15 minutes · Students write every line — no copy-paste allowed

🗣 Say before starting
“We are not learning Python today — that starts in Week 5. The point of this script is to complete the full workflow loop for the first time: create a file in the right place, write code, save, run from the terminal, read the output. Every assignment this semester follows this exact loop.”

“Type every character. Do not copy-paste. Reading and typing code are different cognitive activities. Every character should be deliberate.”
1
Create the file from the terminal
Confirm you are in module01_setup/ first. Then: touch hello_ism3232.py. The file appears in the VS Code Explorer panel. Do not create it by clicking in VS Code. Use the terminal.
2
Open in the editor
code hello_ism3232.py — opens the file in VS Code. Both the terminal and the editor are visible at the same time. This is the working setup for every session.
3
Type the script together — line by line, explain as you go
See the code block below. Pause after the comments section, after the print statements, and after the variables to explain what each part is doing.
4
Save and run from the terminal
⌘S / Ctrl+S to save. Then in the terminal: python3 hello_ism3232.py
hello_ism3232.py — type together, do not copy-paste
# hello_ism3232.py
# ISM3232 – Business Application Development
# Week 1, Module 1: First Python Script
# Author: [Your Name]

# Print a welcome message
print("Hello, ISM3232!")
print("This script is running from the zsh terminal inside VS Code.")

# A variable and a calculation
course_name = "Business Application Development"
credit_hours = 3
weekly_hours = credit_hours * 2

print(f"Course: {course_name}")
print(f"You should expect to spend about {weekly_hours} hours per week.")

print("Environment verified. Week 1 complete.")
Expected output — read this aloud with the class after running
Hello, ISM3232!
This script is running from the zsh terminal inside VS Code.
Course: Business Application Development
You should expect to spend about 6 hours per week.
Environment verified. Week 1 complete.
💬 Debrief discussion — ask after running
Ask: “What just happened? Walk me through exactly what the computer did when you ran that command.”

Look for students to articulate: the shell found the python3 interpreter → the interpreter read the .py file top to bottom → each print() call sent text to standard output → zsh displayed it in the terminal.

If students cannot explain it, walk through it step by step. This understanding — what happens when a script runs — is the foundation for reading tracebacks and debugging in Week 8. Take the time here.
Section 6 — AI Use Policy
6

AI Use Policy Introduction 5 min

Policy overview · Full details in the course outline

🗣 Say
“AI tools are permitted in this course in specific, controlled ways — but not as solution providers. The distinction is this: using AI to understand something is legitimate. Using AI to skip understanding is academic dishonesty.”

“The midterm and capstone presentation both require you to explain your code out loud. If AI wrote it and you cannot explain it, that will be obvious in about thirty seconds.”
✅ Permitted
Explaining an error message in your own setup
Explaining what a command does
Troubleshooting a specific installation failure
Explaining code you have already written
❌ Prohibited
Having AI complete the assignment
Submitting code you cannot explain line by line
Using AI during the midterm or any quiz
Hiding AI use from the instructor
📝
Week 1 specific rule
For Assignment 1: do not use AI at all. The assignment is about verifying that your environment works. There is nothing to explain to an AI. Post in the discussion board or come to office hours if you need help.

Quick Troubleshooting Reference
ProblemFix
WSL not installing (Windows)Run PowerShell as Administrator. If still failing, use a lab machine for Week 1 and troubleshoot with IT before Week 2.
python3 not foundDownload from python.org. On Mac: xcode-select --install installs dev tools including Python.
VS Code terminal not showing zshRe-check the terminal default profile in VS Code settings. Fully restart VS Code, not just the terminal pane.
echo $SHELL returns /bin/bashRun chsh -s /bin/zsh and restart the terminal completely.
code . command not foundIn VS Code: Cmd+Shift+P → “Shell Command: Install ‘code’ command in PATH”. Then restart the terminal.
Permission denied on installOn WSL: prefix the command with sudo. On Mac: do not use sudo with Homebrew commands.
Prompt shows % but not changing to zshAlready running zsh — % is the default zsh prompt character. Run echo $SHELL to confirm.
