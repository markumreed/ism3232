Section 1 — Developer Mindset and Course Identity
1

What is a developer workflow, and why does it matter?

15 minutes · Lecture + discussion · No setup required yet

The core idea to land in Week 1

This course is not a typing class or a memorization course. It is a professional practice course. Every habit you build in Week 1 — how you name files, how you organize folders, how you verify your tools before touching code — is what separates a developer who ships working software from one who spends hours debugging environment problems.

The tools we install today are not obstacles between you and Python. They are the job. A business analyst who can navigate a codebase, run a script, manage dependencies, and push clean work to GitHub is genuinely more valuable than one who cannot — regardless of how well they know Python syntax.

What this course is

A business application development course where students build small business information systems using Python, zsh, GitHub, OOP, SQL, Streamlit, and responsible GenAI workflows.

What this course is not

Not just a programming course. Not just a tools course. Students who treat it as "learn some Python syntax" will struggle on the midterm and capstone, both of which test the full professional workflow.

Discussion prompt 1
"Open your computer right now and find where you saved the last file you worked on. How long did it take? Could you do it from a terminal without clicking anything?"
Goal: surface the fact that most students have never thought about where files live. This is the entry point for explaining paths and directory trees.
Discussion prompt 2
"When a developer says a script 'runs,' what exactly is happening? Who is running it, and from where?"
Goal: distinguish between double-clicking an app vs. executing a script from a shell. Leads into the VS Code terminal and zsh.
Section 2 — Files, Folders, Paths, and Directory Trees
2

How your computer organizes everything

20 minutes · Conceptual lecture · Draw on board or annotate slides

Key mental model: the directory tree

Every file on a computer lives at a specific address called a path. A path is just a sequence of folder names that tells the computer how to find a file, starting from the root of the disk (/ on Mac/Linux, C:\ on Windows) or from wherever you are right now.

Draw this on the board and have students copy it — understanding the tree structure is a prerequisite for every terminal command they will run this semester.

Directory tree concept — draw and annotate with the class
/                          # root of the filesystem (Mac/Linux)
└── Users/
    └── yourname/          # your home directory  ~
        ├── Desktop/
        ├── Documents/
        └── ism3232/       # ← your course folder lives HERE
            ├── module01_setup/
            │   ├── hello_ism3232.py
            │   └── README.md
            ├── module02_zsh/
            ├── module03_git_github/
            ├── module04_programming/
            ├── module05_functions/
            ├── module06_oop/
            ├── module07_final_project/
            ├── data/
            └── screenshots/
Absolute path

The complete address from the root. Always works regardless of where you are. Starts with / on Mac/Linux.

/Users/yourname/ism3232/module01_setup/hello_ism3232.py

Relative path

The address from wherever you are right now. Only works if you are in the right place. Much shorter to type.

If you are in ism3232/: module01_setup/hello_ism3232.py

Symbol / TermWhat it meansExample
/Root of the entire filesystem (Mac/Linux). The top of the tree./Users/yourname
~Your home directory. Shortcut for /Users/yourname.~/ism3232
.The current directory — wherever you are right now../module01_setup
..One directory up from where you are.cd .. goes up one level
File extensionThe suffix after the dot. Tells the OS and editor what type of file it is..py · .md · .txt · .csv
💡
Instructor tip — make this concrete
Ask students to find their Downloads folder in Finder/Explorer, then ask: "What is the full path to that folder?" Most will not know. Then show it in the terminal with echo ~/Downloads. The gap between clicking and knowing where something lives is the core lesson of this section.
Section 3 — VS Code, the Integrated Terminal, and zsh
3

Installing and verifying the required stack

20 minutes · Live demo + student follow-along · Everyone must verify before moving on

🖥
Required tools this week
VS Code — required editor  ·  Python 3.x — required language  ·  Git — required version control  ·  zsh — required shell  ·  WSL Ubuntu — required for Windows students  ·  GitHub account — required  ·  VS Code Python extension — required  ·  VS Code integrated terminal set to zsh — required

Walk through each step below as a live demo. Pause after each step and verify that every student has completed it before proceeding. Do not let anyone skip verification and move ahead.

1
VS Code — install and open
Download from code.visualstudio.com. Install the Python extension (search "Python" in the Extensions panel, install the Microsoft one). Open VS Code and verify the Extensions panel shows Python as installed.
2
Windows only — WSL + Ubuntu + zsh
Open PowerShell as Administrator and run wsl --install. Restart when prompted. Open the Ubuntu app, complete the UNIX user setup, then run sudo apt update && sudo apt install zsh -y and chsh -s $(which zsh). Restart the terminal. This is the most failure-prone step — watch the room carefully.
3
Mac/Linux — verify zsh is the default shell
Mac Catalina and later use zsh by default. Run echo $SHELL — it should return /bin/zsh. If not, run chsh -s /bin/zsh and restart the terminal.
4
Configure VS Code's integrated terminal to use zsh
Open VS Code → ⌘, (Mac) or Ctrl+, (Windows) → search "terminal default profile" → set to zsh (Mac/Linux) or WSL (Windows). Close and reopen the terminal panel. The prompt should show a % or your zsh theme.
5
Verify the full required stack
Run each command and confirm the expected output. Everyone in the room must see all four pass before proceeding.
Verification commands — every student runs these and screenshots the output
# 1. Confirm zsh is running
echo $SHELL
# Expected: /bin/zsh  (or /usr/bin/zsh on some Linux)

# 2. Confirm zsh version
zsh --version
# Expected: zsh 5.x or higher

# 3. Confirm Python 3 is installed
python3 --version
# Expected: Python 3.10 or higher (3.11+ preferred)

# 4. Confirm Git is installed
git --version
# Expected: git version 2.x

# 5. Show current location
pwd
# Expected: your home directory  e.g. /Users/yourname

# 6. List files in current directory
ls
# Expected: Desktop  Documents  Downloads  etc.
✓
Checkpoint — before moving on
Every student should be able to see all six commands returning expected output in the VS Code integrated terminal running zsh. If anyone is still stuck on WSL, pair them with a neighbor or move them to a lab machine. Do not let setup problems prevent participation in the live coding section.
Section 4 — Course Folder Structure
4

Building the course folder — live code-along

10 minutes · Students follow along in their own terminal

Why folder structure matters before Week 2

Students who do not have a consistent, organized course folder spend the rest of the semester hunting for files, making path errors in scripts, and failing to submit the right files. Getting this right in Week 1 takes five minutes. Fixing it in Week 8 takes much longer.

Have every student create this structure today, exactly as shown. The naming conventions (lowercase, underscores, module numbers) are deliberate — they mirror professional Python project naming conventions they will use for the rest of the course.

Live code-along — create the course folder structure from the terminal
# Start from your home directory
cd ~

# Create the top-level course folder
mkdir ism3232
cd ism3232

# Create all module folders at once
mkdir module01_setup module02_zsh module03_git_github \
      module04_programming module05_functions \
      module06_oop module07_final_project \
      data screenshots

# Verify the structure looks right
ls

# Open the entire course folder in VS Code
code .

# Move into the Week 1 module folder
cd module01_setup
💡
Instructor tip — explain code .
code . opens VS Code with the current folder as the workspace. The dot means "here." This is how developers open projects — they navigate to the project folder first, then open the editor. It is different from double-clicking a single file. Emphasize this: the workspace, not the file, is the unit of work.
Section 5 — Writing and Running the First Python Script
5

hello_ism3232.py — live code-along

15 minutes · Students write every line themselves — no copy-paste

The point of this script is not the Python

Students will learn Python properly in Unit 2. The point of hello_ism3232.py is to complete the full workflow loop for the first time: create a file in the right place → write code → save → run from the terminal → verify the output. Every assignment for the rest of the semester follows this same loop.

1
Create the file from the terminal
Make sure you are in ism3232/module01_setup/. Run touch hello_ism3232.py. The file appears in the VS Code Explorer panel. Do not create it by clicking in VS Code — use the terminal.
2
Open and write the script
Run code hello_ism3232.py to open it in the editor. Type the script below — do not paste. Reading and typing code is a different cognitive activity from reading code someone else typed. Every character should be deliberate.
3
Run from the terminal
Save the file (⌘S / Ctrl+S). In the terminal (make sure you are in the same folder as the file), run python3 hello_ism3232.py. Read and discuss the output together.
hello_ism3232.py — type this together, line by line
# hello_ism3232.py
# ISM3232 – Business Application Development
# Week 1, Module 1: First Python Script
# Author: [Your Name]

# Print a welcome message
print("Hello, ISM3232!")
print("This script is running from the zsh terminal inside VS Code.")

# Show some basic Python: a variable and a calculation
course_name = "Business Application Development"
credit_hours = 3
weekly_hours = credit_hours * 2

print(f"Course: {course_name}")
print(f"You should expect to spend about {weekly_hours} hours per week on coursework.")

# Confirm the script completed
print("Environment verified. Week 1 complete.")
Expected terminal output
Hello, ISM3232!
This script is running from the zsh terminal inside VS Code.
Course: Business Application Development
You should expect to spend about 6 hours per week on coursework.
Environment verified. Week 1 complete.
Debrief discussion — after running the script
"What just happened? Walk me through exactly what the computer did when you ran that command."
Look for students to articulate: the shell found the python3 interpreter → the interpreter read the .py file top to bottom → each print() call sent text to standard output → zsh displayed it in the terminal. If they cannot explain it, walk through it slowly. This understanding is the foundation for debugging later.
Section 6 — AI Use Policy Introduction
6

How we use AI in this course — and how we do not

5 minutes · Policy overview · Full details in the course outline

✓ Permitted AI uses
Explain an error message Explain what code does Suggest test cases Draft README sections Explain SQL queries Review class design

AI is a development assistant, not a solution provider. You use it to understand — not to avoid understanding.

✗ Prohibited AI uses
Complete the assignment Submit code you can't explain Skip the debugging process Complete quizzes or exams Hide AI use from instructor

The midterm and capstone presentations both require you to explain your code. If AI wrote it and you cannot explain it, that will be obvious.

💡
Week 1 AI policy — this week specifically
For Assignment 1, do not use AI at all. The assignment is about verifying that your environment works and that you can navigate your own computer. There is nothing to explain to an AI. If you need help, post in the course discussion board or come to office hours.

If something is broken

WSL not installing (Windows): Try from an administrator PowerShell. If it still fails, use a lab machine for Week 1 and troubleshoot with IT.

python3 not found: Download from python.org. On Mac, xcode-select --install installs the dev tools including Python.

VS Code terminal not showing zsh: Re-check the terminal default profile setting. May need to restart VS Code completely.
