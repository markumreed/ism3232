Requirement-by-Requirement Walkthrough
1
All four data types with business variables

Clone or browse the GitHub repo. Open the main Python script. Verify all four types are present.

🔍 What the grader checks
str: at least two string variables with descriptive business names (product_name, status, category — not x, y, s)
int: at least one integer variable (quantity, count, id — not a float stored as int)
float: at least one decimal variable (price, rate, discount — has a decimal point in the value)
bool: derived from a comparison (is_approved = total > 1000 — NOT is_approved = True hardcoded)
✅ Full credit bool
is_over_limit = total > 1000 # derived from comparison → True or False
❌ No credit bool
is_approved = True # hardcoded — not derived from data
⚠ Common mistakes at this step
Boolean is hardcoded as True or False instead of computed from a comparison
Variable names are single letters (x, y, z) — not descriptive business names
int and float present but both are assigned the same value — e.g. quantity = 3.0 (that is a float, not int)
2
Three arithmetic computations

Verify the script computes at least three derived values. Common correct pattern: subtotal, tax, total. Check the math is right.

🔍 What the grader checks
At minimum: multiplication, percentage calculation (multiply by rate), and addition of components
Calculations use the variables defined in Req 1 (not hardcoded numbers in the calculation)
Results are arithmetically correct (spot-check: 450 * 3 = 1350, 1350 * 0.07 = 94.50)
⚠ Common mistakes at this step
Only one calculation (e.g. total = price * quantity) — need three distinct operations
Calculations reference variables not defined earlier in the script
Tax rate applied twice or addition done in wrong order
3
Formatted summary with f-strings

Check the screenshot output. Must have 4+ labelled lines. All currency values must show exactly 2 decimal places.

🔍 What the grader checks
At least 4 lines of output in the terminal screenshot
Currency values formatted with :.2f (1450.50 not 1450.5 or 1450.500)
Each line is labelled (not just raw numbers)
At least one f-string with a computed variable (not just print('hello'))
✅ Correct formatting
Total: $1,444.50 Requires approval: True
❌ Missing .2f
Total: 1444.5 # no decimal, no $ sign, not 2 places
⚠ Common mistakes at this step
:.2f format spec missing — number shows as 1444.5 or 1444.5000000001
Only 2-3 lines of output — need 4+
Lines not labelled — just prints raw numbers
4
input() with immediate type conversion

The input() call must be wrapped in int() or float() on the same line. Check that the converted value is used in a subsequent calculation.

🔍 What the grader checks
input() wrapped in float() or int() on the same line: amount = float(input('...'))
Converted value used in arithmetic — not just stored and printed as-is
Screenshot shows the input prompt appearing and a value typed by the student
Result of the calculation using the input value is printed
✅ Full credit
amount = float(input('Enter amount: ')) new_total = amount * 1.07 print(f'Total: ${new_total:.2f}')
❌ No credit
amount = input('Enter amount: ') print(amount) # just prints the string, no calculation
⚠ Common mistakes at this step
input() stored as string, converted separately later — must be immediate
Input received but not used in any calculation
Screenshot does not show the input prompt — student must run the script interactively
5
Three pytest tests

Open tests/test_week5.py. Verify three functions, each with a meaningful assert.

🔍 What the grader checks
tests/test_week5.py exists in the repo
Three functions each starting with test_
Each asserts a calculated value — not assert True or assert 1 == 1
pytest passes (check ritual screenshot — pytest must show green)
6
Full pre-submission ritual

Check the ritual screenshot for all 9 steps passing.

🔍 What the grader checks
ruff format . and ruff check . — both run, no errors
pytest — all tests green
git commit with descriptive message
git push — push successful
Known Common Mistakes
⚠ Known common mistakes (from the assignment page)
Using input() without converting: age = input(...) then age + 1 — this crashes with a TypeError
Boolean that is always True: is_approved = True — it must be derived from a comparison like total > 1000
Currency displayed without :.2f: printing $1450.6 instead of $1,450.60
Tests that assert obvious constants: assert 1 == 1 — tests must verify your actual calculations

Grading Flow
Clone or open the GitHub repo. Navigate to the main Python script.
Req 1: Read each variable — check str (2+), int (1+), float (1+), bool (computed from comparison).
Req 2: Identify the three arithmetic operations. Spot-check the math manually.
Req 3: Check the screenshot output — count lines, verify :.2f on currency values.
Req 4: Find the input() line — verify float() or int() wraps it. Check that result is used in calculation.
Req 5: Open tests/test_week5.py — count test functions, check assert statements.
Req 6: Check ritual screenshot — all steps present, pytest green, push successful.
Total and return.

Rubric discussion
The four-data-types requirement (15 pts) deserves careful checking because the bool is consistently the most poorly done. Award 12/15 if three types are correct and the bool is hardcoded. Award 10/15 if only two types meet the requirement. The f-string formatting requirement (15 pts) is all-or-nothing on the :.2f spec — if any currency value is not formatted, deduct 5 pts.
