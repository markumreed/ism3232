Section 1 — Project Proposal
Write the proposal before touching any code

The project proposal forces students to think through the full design before a single line of capstone code is written. A student who cannot answer all ten fields does not yet know what they are building — and that uncertainty will hurt every subsequent week.

Proposals are submitted at the end of Week 13 and require instructor approval before the student begins building.

Required project proposal — all 10 fields must be completed
# ISM3232 — Capstone Project Proposal

Project Name:      [e.g., Employee Leave Request Tracker]
Business Problem:  [What problem does this solve? Who has this problem?]
Primary User:      [Who will use this system?]
Records Stored:    [What data is stored? e.g., leave requests, employees]
Business Rules:    [At least 2 rules. e.g., requests over 5 days need director approval]
OOP Classes:       [List your classes and their key attributes + methods]
SQL Tables:        [List your tables and their columns + types]
Streamlit Features:[Add record, view records, filter, update status, one report]
GenAI Feature:     [What will AI do? e.g., summarise a request in plain English]
Out of Scope:      [What are you explicitly NOT building? Keeps scope realistic]

Section 2 — SQL Foundations
SQL is how business data is stored and retrieved at scale

SQLite is the database engine for this course — it stores the entire database in a single .db file with no server required. The SQL you learn here (CREATE TABLE, INSERT, SELECT, WHERE, UPDATE, GROUP BY) applies directly to PostgreSQL, MySQL, and every production database system.

Required SQL operations — all used in the capstone
-- Create a table (run once)
CREATE TABLE requests (
    id        INTEGER PRIMARY KEY,
    requester TEXT    NOT NULL,
    category  TEXT    NOT NULL,
    amount    REAL,
    status    TEXT    DEFAULT 'Pending',
    notes     TEXT
);

-- Insert records
INSERT INTO requests (requester, category, amount)
VALUES ('Taylor', 'Travel', 1200.00);

-- Query all records
SELECT * FROM requests;

-- Filter records
SELECT * FROM requests WHERE status = 'Pending';
SELECT * FROM requests WHERE amount > 1000;

-- Sort results
SELECT * FROM requests ORDER BY amount DESC;

-- Update a record
UPDATE requests SET status = 'Approved' WHERE id = 1;

-- Aggregate reports
SELECT status, COUNT(*), SUM(amount), AVG(amount)
FROM requests
GROUP BY status;
Section 3 — Running SQL in the Terminal with sqlite3
Exploring a SQLite database from the terminal
# Open or create a database file
sqlite3 requests.db

-- Inside the sqlite3 shell:
.tables                         -- list all tables
.schema requests                -- show CREATE TABLE statement
.mode column                    -- formatted column output
.headers on                     -- show column names
SELECT * FROM requests;         -- query the table
.quit                           -- exit the shell
SQL keywordPurposeBusiness example
CREATE TABLEDefine a table and its columnsSet up the requests table once at startup
INSERT INTOAdd a new row to a tableAdd a new purchase request
SELECTRetrieve rows from a tableShow all pending requests
WHEREFilter rows by a conditionRequests where amount > 1000
UPDATE ... SETChange values in existing rowsApprove a request: set status = "Approved"
ORDER BYSort the result setShow requests from largest to smallest amount
GROUP BYAggregate rows by a column valueCount requests by status
COUNT / SUM / AVGCompute a single value across rowsTotal amount of pending requests
