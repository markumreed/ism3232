Section 1 — Search Tools
Why developers need codebase search

As projects grow, you need to find where a function is defined, where a variable is used, or which file contains a specific error. Clicking through VS Code is slow. Knowing rg means answering these questions in seconds from anywhere.

Required search commands — ripgrep, find, grep
# ripgrep (rg) — fast, respects .gitignore
rg 'def '              # find every function definition in the project
rg 'class '            # find every class definition
rg 'TODO'              # find all TODO comments
rg -l 'import'         # list only filenames containing a match
rg 'approve' --type py # search only Python files

# find — locate files by name or type
find . -name '*.py'    # all Python files in current tree
find . -name '*.txt'   # all text files
find . -type d         # all directories

# grep — universal, slower than rg
grep -R 'TODO' .       # recursive search (rg is preferred)

# tree — visual structure
tree -L 3              # directory tree 3 levels deep
💡
Install ripgrep
Mac: brew install ripgrep  ·  Ubuntu/WSL: sudo apt install ripgrep. It is dramatically faster than grep and ignores .venv/ and .git/ automatically because it respects .gitignore.
Section 2 — fzf and zoxide
fzf — fuzzy finder

Interactive fuzzy search for files. Run fzf and type part of a filename — the list narrows in real time. Press Enter to select. Integrate with your shell to fuzzy-search command history with Ctrl+R.

Install: brew install fzf or apt install fzf

zoxide — smarter cd

Remembers directories you visit most. After going to ~/ism3232/module06_oop/ a few times, z oop takes you there from anywhere. Replaces cd for frequently visited paths.

Install: brew install zoxide or apt install zoxide

Section 3 — The Professional Pre-Submission Ritual
📋
Required for every major assignment
Students must run this exact sequence before every submission. It catches formatting errors, linting issues, failing tests, and uncommitted changes before they reach the grader. Instructors check Git logs and Ruff output as part of grading.
Pre-submission ritual — memorize this sequence
# 1. Confirm location and structure
pwd
tree -L 3

# 2. Activate the virtual environment
source .venv/bin/activate

# 3. Format and lint with Ruff
ruff format .        # auto-formats all Python files in place
ruff check .         # reports style and logic issues — fix any errors

# 4. Run all tests
pytest               # all tests must pass — fix failures before proceeding

# 5. Commit and push
git status
git add .
git commit -m 'complete assignment 4'
git push
⚠
Do not submit if pytest fails
If pytest returns failures, fix the code before committing. Committing a broken state and submitting it is not acceptable. The ritual is not complete until all steps pass cleanly.
Section 4 — Git and GitHub Fundamentals
What Git is and why it matters for grading

Git tracks every change to your code — what changed, when, and why. GitHub is where your repositories live and how you submit every major assignment in this course. Instructors see your commit history. A single commit called "final" tells us nothing about your process.

Git first-repository workflow — run in module02_zsh/
# One-time identity setup (if not done in Week 1)
git config --global user.name 'Your Name'
git config --global user.email 'you@email.com'

# Initialize a repository in the current folder
git init
git status                   # see all untracked files

# Create .gitignore before first commit
echo '.venv/' > .gitignore
echo '__pycache__/' >> .gitignore

# Stage and make first commit
git add .
git commit -m 'initial commit: week 4 zsh and workflow module'

# Connect to GitHub repository (create repo on github.com first)
git remote add origin https://github.com/YOURUSERNAME/ism3232-module02.git
git branch -M main
git push -u origin main

# Verify success
git log --oneline    # see commit history
git status           # should say nothing to commit
CommandWhat it doesWhen to run it
git initInitializes a new local repositoryOnce per project, in the project root
git statusShows what has changed and what is stagedBefore every git add
git add .Stages all modified and new filesAfter reviewing git status
git commit -mSaves a snapshot with a messageAt meaningful checkpoints, not just at submission
git log --onelineShows compact commit historyTo verify your commits were recorded
git pushSends commits to GitHubFinal step of the pre-submission ritual
