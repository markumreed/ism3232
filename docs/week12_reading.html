The four OOP mistakes to catch this week

All four mistakes — cause each error, then fix it
# Mistake 1: missing self parameter
def approve():            # ← wrong: TypeError on call
def approve(self):        # ← correct

# Mistake 2: local variable instead of self.attribute
    status = 'Approved'      # ← wrong: vanishes after method returns
    self.status = 'Approved' # ← correct

# Mistake 3: print() instead of return
    print(self.amount > 1000)   # ← wrong: cannot assert a print
    return self.amount > 1000   # ← correct

# Mistake 4: class-level list
class Manager:
    items = []             # ← wrong: shared by all instances
    def __init__(self):
        self.items = []    # ← correct

Minimum five tests required

  • Default state test — status is Pending after instantiation
  • Method change test — approve() sets status to Approved
  • Boolean True test — requires_review() returns True above the limit
  • Boolean False test — requires_review() returns False below the limit
  • Instance independence test — approving one does not affect others

A second domain — see the same pattern differently

One of the most common struggles in Week 12 is designing a class for a new domain when every example you have seen uses purchase requests. Here is the same two-class manager pattern applied to an employee leave system — so you can see how the structure transfers.

LeaveRequest + LeaveManager — same pattern, different domain
# design.md (write this first, commit before code)
#
# Entity class: LeaveRequest
# Attributes: req_id (int), employee (str), start_date (str),
#             num_days (int), reason (str), status (str, default 'Pending')
# Methods:
#   approve() -> None
#   reject() -> None
#   requires_director_review() -> bool  (num_days > 10)
#   __repr__() -> str
#
# Manager class: LeaveManager
# Attributes: requests (list, instance-level)
# Methods:
#   add_request(req) -> None
#   list_by_status(status) -> list
#   total_days_pending() -> int
#   summary_report() -> None

DIRECTOR_THRESHOLD = 10

class LeaveRequest:
    """Represents one employee leave request."""

    def __init__(self, req_id: int, employee: str,
                 start_date: str, num_days: int, reason: str) -> None:
        self.req_id     = req_id
        self.employee   = employee
        self.start_date = start_date
        self.num_days   = num_days
        self.reason     = reason
        self.status     = 'Pending'

    def approve(self) -> None:
        self.status = 'Approved'

    def reject(self) -> None:
        self.status = 'Rejected'

    def requires_director_review(self) -> bool:
        return self.num_days > DIRECTOR_THRESHOLD

    def __repr__(self) -> str:
        return (f'LeaveRequest(id={self.req_id}, employee={self.employee!r}, '
                f'days={self.num_days}, status={self.status!r})')


class LeaveManager:
    """Manages a collection of LeaveRequest objects."""

    def __init__(self) -> None:
        self.requests = []   # instance-level — not class-level

    def add_request(self, req: LeaveRequest) -> None:
        self.requests.append(req)

    def list_by_status(self, status: str) -> list:
        return [r for r in self.requests if r.status == status]

    def total_days_pending(self) -> int:
        return sum(r.num_days for r in self.list_by_status('Pending'))

    def summary_report(self) -> None:
        pending  = self.list_by_status('Pending')
        approved = self.list_by_status('Approved')
        print(f'Total requests:  {len(self.requests)}')
        print(f'Pending:         {len(pending)} ({self.total_days_pending()} days)')
        print(f'Approved:        {len(approved)}')
The pattern is identical

Compare LeaveRequest to BusinessRequest. The structure — __init__ with default status, two status methods, one boolean business rule, __repr__ — is identical. The manager — instance-level list, add, list_by_status, aggregate, report — is identical. When you design your own capstone class, start here. Replace the attributes and the business rule threshold. The scaffolding is the same.
