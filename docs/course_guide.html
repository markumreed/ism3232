INSTRUCTOR RESOURCE · COURSE DESIGN

Course Assembly Guide

ISM3232 is built as a library of 14 teaching weeks plus a midterm and a capstone week, for 16 weeks total. Professors can assemble any subset into a 6-week, 9-week, or full 16-week course without rewriting anything — choose your weeks and assign them in order.

How this works

Each week contains four components: a Reading page (concepts and quiz), a Lecture page (worked examples and in-class activities), a Lab page (hands-on exercises), and Slides (for in-person sessions). You can assign all four or only what your format supports. An async course might rely on Reading and Lab only; a hybrid session might use Lecture and Lab in class with Slides projected.

Week Catalog

There are 14 teaching weeks plus one midterm week and one capstone week, for 16 weeks total. Each week is a complete, self-contained lesson. The table below shows every week, its key topics, approximate time, and which weeks should come before it.

Can OOP start earlier?

Yes. In compressed formats, you can begin W10 (Classes) immediately after W07 (Functions), skipping W08's debugging review and replacing W09's formal midterm with an informal progress check. OOP only requires that students can write and call functions — the debugging and AI literacy material in W08 can follow OOP rather than precede it.

6-Week Course — Intensive / Workshop Format

Designed for boot-camp sessions, corporate training, or an accelerated summer intensive. Each "week" is a full day (6–8 hours) or two 3-hour sessions. Six blocks cover all essential skills; the demo is a focused presentation on the final day.

Format assumptions

Students meet for full-day sessions (6–8 hours) or twice-weekly for 3 hours per session. The formal midterm is replaced with an ungraded progress check. Git is taught in Block 1 rather than isolated — students commit every day from the start.

Assessment — 6-Week Format

  • Block quizzes (2× informal) — 5-question ungraded checks at end of Blocks 2 and 4. Used for pacing, not graded.
  • Lab submissions (each block) — students submit the lab from each block. Graded on completion + correctness (does the code run, does it produce expected output).
  • Mini Demo (Block 6) — 3-minute live presentation using the capstone rubric compressed to: one user story, one database table, one Streamlit page, one AI interaction.
What to skip and why

W08 Debugging Review: replace with a 30-minute code trace quiz — you get diagnostic data without losing a full session.
W09 Formal Midterm: skip entirely. An informal check at the end of Block 3 achieves the same diagnostic goal.
Full OOP design patterns: compress W12 to an afternoon lab rather than a full day. Students get the practice without all the theory.

9-Week Course — Trimester / Accelerated Semester

Designed for university trimesters, 10-week quarters (with finals week off), or half-semester intensive electives. Each "week" is one standard class week (2–3 class sessions, 4–6 total hours). All core concepts are covered; the formal midterm is replaced with a formative assessment.

Format assumptions

Two or three class sessions per week, each 75–90 minutes. Labs are completed in class or assigned as take-home (1–2 hours). Students should have their developer environment working by the end of Week 1 — do not let environment issues linger into Week 2.

Assessment — 9-Week Format

  • Weekly labs (8×) — each submitted via GitHub. Graded on the standard lab rubric. Collectively 40% of grade.
  • Formative midpoint check (Week 4) — 30-minute ungraded in-class code trace quiz covering W01–W07. Replaces the formal midterm; gives the instructor a read on student readiness before OOP. Not counted toward final grade but students must attempt it.
  • Capstone demo (Week 9) — 5-minute live presentation using the full capstone rubric. Minimum requirements: one user story implemented end-to-end, one database table, one Streamlit page, one working AI interaction.

16-Week Course — Full Semester

The original design. One week per topic, with a midterm exam at Week 9. Each week has three class sessions: Reading (assigned before Tuesday), Lecture (Tuesday), Lab (Thursday), and Slides available for in-person sessions. The course ends with a full demo-day presentation in Week 16.

Tips for Combining Weeks

When you merge two weeks into one class session, these pairings work best:

W01 + W02 (Setup + zsh)

First session: install everything, run first script. Second session: file commands and navigation lab. Share the 9-step ritual across both — it reinforces the "this is how you start every work session" habit from day one.

W03 + W04 (venv + Git)

Morning: create a venv, install packages, write requirements.txt. Afternoon: git init the project, add everything, push to GitHub. Students see the full environment setup workflow as one continuous process.

W05 + W06 (Types + Control Flow)

Types give you data to work with; control flow gives you ways to make decisions with it. Teach variables and operators in the morning, then conditionals, loops, and dictionaries in the afternoon on the same business dataset (a product catalog).

W10 + W11 (Classes + Composition)

Define a class in the morning; in the afternoon, build a second class that owns an instance of the first. Students see exactly why composition exists — not as a concept, but as a solution to a real problem they just hit.

W13 + W14 (SQL + Python SQL)

Design the schema and write raw SQL in the morning; wrap it in Python functions in the afternoon. Students stay on the same database, so there's no context switch — just a cleaner interface layer appearing around their existing SQL.

W15 + W16 (Streamlit + Demo)

Build the interface Monday through Wednesday; integrate the AI feature Thursday; demo Friday. The compressed timeline forces scope decisions that teach real project management — "what's the minimum that makes this feel like an app?"

What to Skip (and What to Replace It With)

Skipping the Formal Midterm (W09)

What you lose: A summative checkpoint halfway through the semester.
What to do instead: Replace with a 30-minute in-class code trace exercise at the end of W08. Students trace three short functions and explain two bugs. Grade it pass/fail. You keep the diagnostic value without the prep overhead.

Skipping W08 (Debugging Review)

What you lose: Dedicated time on tracebacks, the debugger, and AI literacy discussion.
What to do instead: Integrate the AI literacy section into W07 (Functions) as a 20-minute discussion. Cover tracebacks as they arise in W10–W12 labs — students encounter real errors during OOP and that's the best moment to teach debugging.

Skipping W12 (OOP Applied)

What you lose: The end-to-end OOP mini-project and capstone planning session.
What to do instead: Move the capstone schema design into W13's first session. Skip the OOP mini-project; students will get applied OOP practice in W13–W14 as they model their database entities. The compressed version loses some design iteration but gains a week of build time.

Choosing the Right Format

If you're uncertain which format to use:

  • Corporate training or professional development → 6-week intensive. Skip the formal midterm. Focus on the database + Streamlit stack — that's what practitioners want to take back to work.
  • 10-week quarter or trimester → 9-week course. Every concept is covered at a pace that allows students to actually build something. The week of buffer before finals is your overflow/catch-up session.
  • 16-week semester with general business students → Full course as designed. The pacing allows time for environment confusion in W01–W02, which is when most non-technical students get stuck.
  • Graduate elective or MBA module → Start at W05 (skip developer foundations), move quickly through W05–W08, spend full time on W10–W16. Graduate students often already have a working environment; the value is in OOP, SQL, Streamlit, and the AI integration.
Building your own sequence

The hard constraint is: environment before Python (W01–W04 before W05+), Python functions before OOP (W07 before W10), OOP before database (W12 before W13), database before Streamlit (W14 before W15), Streamlit before GenAI (W15 before W16). Everything else is flexible. W08 (debugging review) and W09 (midterm) are the easiest to compress or drop. W03 (venv) and W04 (Git) can be combined into a single session in compressed formats.
