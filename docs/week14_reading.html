The connection pattern — memorise this

with sqlite3.connect() as conn

The with statement is a context manager. When the block exits — normally or due to an error — the connection is automatically committed and closed. Every database function uses this pattern.

The required pattern and why ? placeholders are mandatory
import sqlite3

DB_FILE = 'data/requests.db'

def some_function(value, db_file=DB_FILE):
    with sqlite3.connect(db_file) as conn:
        conn.execute(
            'SQL WITH ? PLACEHOLDERS',
            (value,)   # values as a tuple
        )

# NEVER do this:
conn.execute(f'INSERT INTO t VALUES ({value})')  # SQL injection risk

# ALWAYS do this:
conn.execute('INSERT INTO t VALUES (?)', (value,))
⚠ f-strings in SQL are dangerous

An f-string in a SQL statement allows any string — including malicious SQL commands — to become part of the query. ? placeholders escape the value automatically. Every SQL statement in this course uses ? — no exceptions.

All five required functions

database.py — type every line
import sqlite3
import os

DB_FILE = 'data/requests.db'

def create_table(db_file=DB_FILE):
    os.makedirs(os.path.dirname(db_file), exist_ok=True)
    with sqlite3.connect(db_file) as conn:
        conn.execute('''
            CREATE TABLE IF NOT EXISTS requests (
                id        INTEGER PRIMARY KEY AUTOINCREMENT,
                requester TEXT    NOT NULL,
                category  TEXT    NOT NULL,
                amount    REAL,
                status    TEXT    DEFAULT 'Pending'
            )
        ''')

def add_record(requester, category, amount, db_file=DB_FILE):
    with sqlite3.connect(db_file) as conn:
        conn.execute(
            'INSERT INTO requests (requester, category, amount) VALUES (?,?,?)',
            (requester, category, amount)
        )

def get_all_records(db_file=DB_FILE):
    with sqlite3.connect(db_file) as conn:
        conn.row_factory = sqlite3.Row
        cur = conn.execute('SELECT * FROM requests ORDER BY id DESC')
        return [dict(row) for row in cur.fetchall()]

def update_status(record_id, new_status, db_file=DB_FILE):
    with sqlite3.connect(db_file) as conn:
        conn.execute('UPDATE requests SET status=? WHERE id=?',
                     (new_status, record_id))

def get_status_report(db_file=DB_FILE):
    with sqlite3.connect(db_file) as conn:
        cur = conn.execute('''
            SELECT status, COUNT(*) AS count, SUM(amount) AS total
            FROM requests GROUP BY status
        ''')
        return cur.fetchall()
✍ Think before reading on

Why is conn.execute(f"SELECT * FROM requests WHERE id = {user_id}") dangerous even if user_id looks like a safe integer? What could a user pass as user_id to cause damage?

row_factory = sqlite3.Row

Without row_factory, sqlite3 returns tuples: row[2] for amount. With row_factory, you get Row objects: row['amount']. dict(row) converts each Row to a plain Python dict — the format Streamlit expects for st.dataframe().

Testing with tmp_path

tests/test_database.py — every test gets a fresh isolated database
from database import create_table, add_record, get_all_records, update_status

def test_add_and_retrieve(tmp_path):
    db = str(tmp_path / 'test.db')   # fresh per test
    create_table(db)
    add_record('Taylor', 'Equipment', 1200.00, db_file=db)
    records = get_all_records(db_file=db)
    assert len(records) == 1
    assert records[0]['requester'] == 'Taylor'

def test_default_status(tmp_path):
    db = str(tmp_path / 'test.db')
    create_table(db)
    add_record('Jordan', 'Travel', 350.00, db_file=db)
    records = get_all_records(db_file=db)
    assert records[0]['status'] == 'Pending'

def test_update_status(tmp_path):
    db = str(tmp_path / 'test.db')
    create_table(db)
    add_record('Morgan', 'Software', 2800.00, db_file=db)
    records = get_all_records(db_file=db)
    update_status(records[0]['id'], 'Approved', db_file=db)
    updated = get_all_records(db_file=db)
    assert updated[0]['status'] == 'Approved'
⚠ Every test must pass db_file=db

If any test uses the default DB_FILE, it writes to your development database, not the isolated test database. The tests then depend on each other's data — fragile and unpredictable.

Putting it all together — a complete worked example

Here is the full sequence: create the table, add three records with different statuses, retrieve them as dicts, update one, and run the status report. Run this end-to-end to verify your five functions work together before writing any Streamlit code.

Full integration test — run from the project root
import os
os.makedirs('data', exist_ok=True)

from database import (create_table, add_record, get_all_records,
                      update_status, get_status_report)

# Start fresh
import os
if os.path.exists('data/test_run.db'):
    os.remove('data/test_run.db')

DB = 'data/test_run.db'

# 1. Create table
create_table(DB)

# 2. Insert three records
id1 = add_record('Taylor',  'Equipment', 1200.00, 'Standing desk', db_file=DB)
id2 = add_record('Jordan',  'Travel',     350.00, 'Client visit',  db_file=DB)
id3 = add_record('Morgan',  'Software',  2800.00, 'Licence renewal', db_file=DB)

# 3. Retrieve all — should be a list of dicts
records = get_all_records(db_file=DB)
assert len(records) == 3
assert records[0]['requester'] == 'Morgan'   # newest first
print(f"Retrieved: {len(records)} records")
print(f"First record keys: {list(records[0].keys())}")

# 4. Update one status
update_status(id1, 'Approved', db_file=DB)
updated = get_all_records(db_file=DB)
by_id = {r['id']: r for r in updated}
assert by_id[id1]['status'] == 'Approved'
assert by_id[id2]['status'] == 'Pending'   # unchanged
print("Status update verified")

# 5. Status report
report = get_status_report(db_file=DB)
for row in report:
    print(f"  {row[0]}: {row[1]} requests, ${row[2]:,.2f} total")

print("All integration checks passed.")
⚠ Run this before wiring Streamlit

If any assertion fails, fix the database function before adding the UI layer. A broken database function under a working Streamlit tab is very difficult to debug. Always verify the data layer in isolation first.

Q1 · Multiple Choice
What is the purpose of using ? placeholders in a sqlite3 query?
  • a) To mark optional columns
  • b) To prevent SQL injection by separating SQL structure from user-supplied values
  • c) To match column types automatically
  • d) To create indexes
Answer
Never build SQL with f-strings. conn.execute('INSERT INTO t VALUES (?)', (value,)) is safe; f'INSERT INTO t VALUES ({value})' is a SQL injection vulnerability.
Q2 · Multiple Choice
What does with sqlite3.connect(db_file) as conn: ensure?
  • a) The database is always created fresh
  • b) The connection is committed and closed automatically when the block exits
  • c) All queries run in read-only mode
  • d) The database is locked for the test
Answer
The context manager handles cleanup. If an exception is raised, changes are rolled back. This prevents connection leaks.
Q3 · Code Trace
What does this function return for a new, empty database?
def get_all_records(db_file=DB_FILE):
    with sqlite3.connect(db_file) as conn:
        rows = conn.execute('SELECT * FROM requests').fetchall()
    return rows
Answer
An empty list [] — fetchall() returns all matching rows as a list of tuples. If the table has no rows, it returns an empty list.
Q4 · Multiple Choice
Why do ISM3232 database functions accept a db_file parameter with a default value?
  • a) To switch between MySQL and SQLite
  • b) To let tests use a temporary database instead of the production file
  • c) To set the connection timeout
  • d) To specify the SQL dialect
Answer
Tests pass tmp_path / 'test.db' as db_file, keeping test data completely separate from the real database.
Q5 · Multiple Choice
What does tmp_path provide in a pytest test?
  • a) A mock of the database connection
  • b) A temporary directory unique to the test, cleaned up after the test runs
  • c) A pre-populated test database
  • d) A path to the production database
Answer
Using tmp_path / 'test.db' as db_file ensures each test gets a fresh, isolated database with no leftover data from other tests.
Q6 · Multiple Choice
What does conn.execute() return for an INSERT statement?
  • a) The inserted row as a dict
  • b) A cursor object (use .lastrowid to get the new id)
  • c) The total number of rows in the table
  • d) True if successful, False if not
Answer
The cursor has useful properties: .lastrowid (id of the just-inserted row) and .rowcount (number of rows affected).
Q7 · True / False
You should build SQL query strings using f-strings when values come from user input.
Answer
False — Building SQL with f-strings is a SQL injection vulnerability. Always use parameterised queries with ? placeholders: conn.execute('SELECT * FROM t WHERE id = ?', (id,)).
Q8 · True / False
fetchone() raises an exception if no rows match the query.
Answer
False — fetchone() returns None if no rows match. Always check for None before accessing fields, or you will get TypeError: 'NoneType' object is not subscriptable.
Q9 · Short Answer
What are the five required database functions in the ISM3232 capstone, and what does each one do?
Answer
create_table (creates the table if not exists), add_record (INSERT a new row, return new id), get_all_records (SELECT * and return all rows), update_status (UPDATE a specific row's status by id), get_status_report (SELECT grouped by status and return counts or totals). Each is tested separately.
