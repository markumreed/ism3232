Section 1 — Reading Tracebacks
A traceback is not failure — it is information

Read tracebacks bottom-up: the last line is the error type and message. The lines above show how the program reached that point. Students who read tracebacks carefully fix bugs in minutes. Students who ignore them and immediately ask AI spend hours on the same bugs.

Traceback anatomy — read from bottom to top
Traceback (most recent call last):
  File "main.py", line 14, in <module>
    result = process_request(req)           # called here
  File "business_rules.py", line 8, in process_request
    total = req["amount"] * req["quantty"]  # error on this line
KeyError: 'quantty'                         # THE ACTUAL ERROR: typo in key name

# Reading order:
# 1. KeyError: 'quantty'         — what went wrong
# 2. business_rules.py line 8   — where it happened
# 3. main.py line 14            — what triggered the broken code
Error typeWhat it meansCommon cause
SyntaxErrorPython cannot parse the code at allMissing colon, unclosed parenthesis, bad indentation
NameErrorA name is used before it is definedTypo in variable name, variable used outside its scope
TypeErrorWrong type used in an operationAdding str and int, calling a non-callable
KeyErrorDictionary key does not existTypo in key name, wrong capitalisation
IndexErrorList index out of rangeOff-by-one error, accessing an empty list
ValueErrorRight type but wrong valueint("hello"), float("abc")
Section 2 — Debugging with print()
Strategic print() statements reveal the gap between expectation and reality

Debugging is finding where the program stops doing what you expect. print() lets you inspect values at specific execution points. Start from the error line and work backwards, printing variables until you find where the data became wrong.

Debugging workflow — add prints, run, read, remove when fixed
def process_requests(requests):
    total = 0
    for req in requests:
        print(f"DEBUG: req = {req}")                        # see each record
        amount = req["amount"]
        print(f"DEBUG: amount = {amount} ({type(amount).__name__})") # check type
        total += amount
    print(f"DEBUG: total = {total}")                    # verify accumulation
    return total
# After the bug is fixed, remove all DEBUG lines before ruff + commit
Section 3 — Required AI Literacy Activity: Debug First, Then Ask
★
Graded on process, not just the fixed code
Students who submit only the corrected file with no reflection receive no credit for the activity portion. The six-step written reflection is the assignment.
Required workflow — follow every step in order
Step 1: Read the traceback. Write what you think is wrong in your own words.
Step 2: Try to fix it yourself using print() statements.
        Document what you tried and what you observed.
Step 3: If still stuck, ask AI to explain the error message ONLY.
        Prompt: "Explain this Python error without giving me corrected code."
        Do NOT ask AI to fix the code.
Step 4: Fix the code yourself based on the AI explanation.
Step 5: Run pytest — all tests must pass before continuing.
Step 6: Write reflection.md covering:
        — What was the bug?
        — What did AI explain (paste the AI response)?
        — What did you change and why?
        — How did you verify the fix?
        — Can you explain every line of the fixed code without AI help?
⚠
The prohibited prompt this week
"Fix this code for me." This bypasses the entire learning process and produces code the student cannot explain. It will be obvious on the midterm and capstone presentation.
