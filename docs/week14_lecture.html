Section 1 — Connecting Python to SQLite
Why Python drives the database

SQL is powerful for querying and storing data, but a business application needs Python to provide the logic layer: validate inputs, apply business rules, call the right SQL at the right time, and format output for users. The sqlite3 module is Python's built-in SQLite driver — no installation required.

The pattern: Python opens a connection, creates a cursor, executes SQL, fetches results, closes the connection. Every database function in your capstone follows this exact pattern.

The sqlite3 connection pattern — memorise this
import sqlite3

# Always use a context manager (with) — auto-commits and closes
def get_connection():
    return sqlite3.connect("requests.db")

# Reading data
with get_connection() as conn:
    cursor = conn.cursor()
    cursor.execute("SELECT * FROM requests WHERE status = ?", ("Pending",))
    rows = cursor.fetchall()
    for row in rows:
        print(row)

# Writing data — conn auto-commits when the with block exits
with get_connection() as conn:
    conn.execute(
        "INSERT INTO requests (requester, category, amount) VALUES (?, ?, ?)",
        ("Taylor", "Travel", 1200.0)
    )
⚠
Always use parameterised queries — never f-strings in SQL
Never build SQL strings by concatenating user input: f"WHERE name = '{name}'" is a SQL injection vulnerability. Always use ? placeholders and pass values as a tuple. This applies to every SQL statement in the capstone.
Section 2 — Required database.py Functions
All five functions are required in the capstone

Every capstone submission must have a database.py module containing these five functions. Each function is self-contained: it opens its own connection, does one job, and closes. Functions that need each other pass data by return value, not by sharing connections.

database.py — all five required functions
import sqlite3

DB_FILE = "data/requests.db"

def create_table():
    """Create the requests table if it does not already exist."""
    with sqlite3.connect(DB_FILE) as conn:
        conn.execute("""
            CREATE TABLE IF NOT EXISTS requests (
                id        INTEGER PRIMARY KEY AUTOINCREMENT,
                requester TEXT    NOT NULL,
                category  TEXT    NOT NULL,
                amount    REAL,
                status    TEXT    DEFAULT 'Pending',
                notes     TEXT
            )
        """)

def add_record(requester, category, amount, notes=""):
    """Insert a new request record."""
    with sqlite3.connect(DB_FILE) as conn:
        conn.execute(
            "INSERT INTO requests (requester,category,amount,notes) VALUES(?,?,?,?)",
            (requester, category, amount, notes)
        )

def get_all_records():
    """Return all records as a list of dicts."""
    with sqlite3.connect(DB_FILE) as conn:
        conn.row_factory = sqlite3.Row
        cursor = conn.execute("SELECT * FROM requests ORDER BY id DESC")
        return [dict(row) for row in cursor.fetchall()]

def update_status(record_id, new_status):
    """Update the status of a request by ID."""
    with sqlite3.connect(DB_FILE) as conn:
        conn.execute(
            "UPDATE requests SET status = ? WHERE id = ?",
            (new_status, record_id)
        )

def get_status_report():
    """Return count and total amount grouped by status."""
    with sqlite3.connect(DB_FILE) as conn:
        cursor = conn.execute(
            "SELECT status, COUNT(*) as count, SUM(amount) as total FROM requests GROUP BY status"
        )
        return cursor.fetchall()
Section 3 — row_factory and Working with Results
sqlite3.Row makes results easier to work with

By default, fetchall() returns plain tuples: (1, "Taylor", "Travel", 1200.0, "Pending", ""). Setting conn.row_factory = sqlite3.Row makes each row behave like a dictionary — you access fields by name instead of index. Wrapping in dict(row) gives you a plain Python dict, which Streamlit and your OOP layer can consume directly.

Converting rows to dicts — use in get_all_records()
# Without row_factory — tuple access, fragile
row = (1, "Taylor", "Travel", 1200.0, "Pending", "")
print(row[1])   # Taylor — breaks if columns reorder

# With row_factory — dict access, robust
conn.row_factory = sqlite3.Row
row = conn.execute("SELECT * FROM requests WHERE id=1").fetchone()
record = dict(row)
print(record["requester"])   # Taylor
print(record["amount"])      # 1200.0
Section 4 — Testing Database Functions
Use a test database, not your development database

Never run pytest against data/requests.db — tests will insert garbage data or delete real records. Use a separate test database (e.g. data/test_requests.db) and configure your database module to accept a file path parameter, or use an in-memory SQLite database for tests.

Testing with an in-memory database
import sqlite3, pytest
from database import create_table, add_record, get_all_records, update_status

def test_add_and_retrieve(tmp_path):
    # tmp_path is a pytest fixture — creates a temp folder per test
    db = tmp_path / "test.db"
    create_table(db)
    add_record("Taylor", "Travel", 1200, db_file=db)
    records = get_all_records(db_file=db)
    assert len(records) == 1
    assert records[0]["requester"] == "Taylor"

def test_update_status(tmp_path):
    db = tmp_path / "test.db"
    create_table(db)
    add_record("Jordan", "Equipment", 450, db_file=db)
    records = get_all_records(db_file=db)
    update_status(records[0]["id"], "Approved", db_file=db)
    updated = get_all_records(db_file=db)
    assert updated[0]["status"] == "Approved"
