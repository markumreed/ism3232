OOP Checkpoint before the capstone
Week 12 is the OOP applied practice week. Students build a standalone multi-class simulation — no database, no Streamlit — and write comprehensive pytest tests. This is the last chance to confirm OOP fluency before SQL and Streamlit are introduced on top of it in Unit 4.
Section 1 — What Applied OOP Practice Means
The goal this week: build without a net

Weeks 10 and 11 introduced the concepts with guided examples. This week students design and build their own multi-class system from a prompt — choosing their own class names, attributes, and methods — and test every piece of business logic with pytest.

The deliverable is a Python project with at least two entity classes, one manager class, and five or more pytest tests. No database. No Streamlit. Pure OOP.

Section 2 — Design-First Approach
Always design before you type

Before writing any code this week, students should write down: What are my two entity classes? What attributes does each one have? What methods does each one need? What does the manager class hold and what can it do?

A class design written in comments or a README takes five minutes and saves an hour of refactoring. This is the habit that separates developers who ship from developers who spin.

Section 3 — Reference: The Complete Pattern
Full example: Employee + Department + DepartmentManager

Use this as a model if students are stuck on what to build. They should NOT copy it — they must design their own domain — but they can reference the structure.

models.py — two entity classes + one manager
class Employee:
    def __init__(self, emp_id, name, department, salary):
        self.emp_id     = emp_id
        self.name       = name
        self.department = department
        self.salary     = salary
        self.active     = True

    def give_raise(self, percent):
        self.salary *= (1 + percent / 100)

    def deactivate(self):
        self.active = False

    def is_senior(self):
        return self.salary > 80000

class Department:
    def __init__(self, name, budget):
        self.name       = name
        self.budget     = budget
        self.employees  = []

    def hire(self, employee):
        self.employees.append(employee)

    def headcount(self):
        return len([e for e in self.employees if e.active])

    def payroll_total(self):
        return sum(e.salary for e in self.employees if e.active)

    def over_budget(self):
        return self.payroll_total() > self.budget
tests/test_models.py — five required test minimum
from models import Employee, Department

def test_give_raise_increases_salary():
    e = Employee(1, "Alex", "Eng", 70000)
    e.give_raise(10)
    assert e.salary == 77000

def test_deactivate_removes_from_headcount():
    dept = Department("Eng", 300000)
    e = Employee(2, "Sam", "Eng", 60000)
    dept.hire(e)
    e.deactivate()
    assert dept.headcount() == 0

def test_payroll_excludes_inactive():
    dept = Department("Eng", 300000)
    e1 = Employee(3, "Jo", "Eng", 50000)
    e2 = Employee(4, "Pat", "Eng", 55000)
    dept.hire(e1); dept.hire(e2)
    e2.deactivate()
    assert dept.payroll_total() == 50000

def test_over_budget_true():
    dept = Department("Mktg", 100000)
    dept.hire(Employee(5, "Lee", "Mktg", 120000))
    assert dept.over_budget() is True

def test_is_senior_boundary():
    assert Employee(6, "Kim", "Eng", 80000).is_senior() is False
    assert Employee(7, "Kim", "Eng", 80001).is_senior() is True
