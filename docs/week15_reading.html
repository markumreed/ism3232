The Streamlit execution model — the single most important concept

Every interaction reruns the entire script

When a user clicks a button, Streamlit runs your entire app.py from line 1 to the last line. Not just the button handler. The whole file. Every line. Every time. This explains why create_table() goes at the top outside any tab — it must run on every rerun before any widget reads the database.

Correct app.py structure
import streamlit as st
from database import create_table, add_record, get_all_records, \
                     update_status, get_status_report

# MUST be at the top — runs before any widget, every rerun
create_table()

st.set_page_config(page_title='Purchase Requests', layout='wide')
st.title('Purchase Request Tracker')

tab1, tab2, tab3, tab4, tab5 = st.tabs(
    ['Submit', 'View All', 'Filter', 'Update Status', 'Report'])
Run the app
streamlit run app.py

The five required features

Tab 1 — Submit with validation
with tab1:
    st.subheader('New Purchase Request')
    requester = st.text_input('Your name')
    category  = st.selectbox('Category', ['Equipment','Travel','Software','Other'])
    amount    = st.number_input('Amount ($)', min_value=0.01, format='%.2f')
    if st.button('Submit Request'):
        if requester and amount > 0:
            add_record(requester, category, amount)
            st.success(f'Request submitted: ${amount:,.2f}')
        else:
            st.error('Name and amount are required.')
Tab 3 — Filter (applied in Python, not SQL)
with tab3:
    status_filter = st.selectbox('Filter by status',
                                 ['All','Pending','Approved','Rejected'])
    records = get_all_records()
    if status_filter != 'All':
        records = [r for r in records if r['status'] == status_filter]
    if records:
        st.dataframe(records, use_container_width=True)
    else:
        st.info(f'No {status_filter} requests.')
    st.caption(f'{len(records)} record(s) shown')
Tab 4 — Update Status (selectbox from live database)
with tab4:
    records = get_all_records()   # live — not hardcoded IDs
    if records:
        options = {
            f"ID {r['id']}: {r['requester']} — ${r['amount']:,.2f}": r['id']
            for r in records
        }
        sel = st.selectbox('Select request', list(options.keys()))
        new_status = st.radio('New status',
                              ['Pending','Approved','Rejected'], horizontal=True)
        if st.button('Update'):
            update_status(options[sel], new_status)
            st.success(f'Updated to {new_status}')
            st.rerun()
⚠ Build the selectbox from get_all_records()

Never hardcode IDs in the Update Status tab. If a record is deleted, hardcoded IDs point to the wrong rows. Always build the options dict from live database data.

What Streamlit actually does — the execution model in detail

Streamlit's execution model is unlike any framework most students have used before. Understanding it prevents the majority of Streamlit bugs.

When a user clicks a button, Streamlit does not just run the button's handler. It runs your entire app.py file, from the first line to the last. Every widget is re-evaluated. Every function call executes again. Every variable is set from scratch.

Why create_table() must be at the top
# Correct — create_table runs on every rerun, before any widget
create_table()                          # idempotent: IF NOT EXISTS
st.set_page_config(page_title='App')
tab1, tab2 = st.tabs(['Submit', 'View'])

with tab1:
    # ... submission form ...

with tab2:
    # ... get_all_records() is called HERE, every rerun
    records = get_all_records()         # always fresh from DB
    st.dataframe(records)

If create_table() were inside tab2, it would only run when the user clicks tab 2. If they submit a form on tab 1 first, the INSERT would fail because the table does not exist yet. This is the bug that create_table() at the top prevents.

Widget return values and state

Every widget returns a value on every rerun. st.text_input('Name') returns whatever the user last typed — or an empty string on first load. st.selectbox('Status', options) returns the currently selected option. You do not need event handlers. You just read the return value and act on it.

Reading widget values — no event handlers needed
requester = st.text_input('Your name')   # returns current text
category  = st.selectbox('Category',
                         ['Equipment', 'Travel', 'Software'])
amount    = st.number_input('Amount', min_value=0.01, format='%.2f')

if st.button('Submit'):
    # This block only runs when the button is clicked
    if requester and amount > 0:
        add_record(requester, category, amount)
        st.success(f'Request submitted: ${amount:,.2f}')
    else:
        st.error('Name and amount are required.')

Building the Update Status tab correctly

The most common mistake in Tab 4 is hardcoding record IDs. A hardcoded ID breaks the moment any record is deleted or the database is reset. Always build the selectbox options from a live call to get_all_records().

Update tab — live selectbox from database
with tab4:
    records = get_all_records()   # live — reflects current DB state
    if records:
        # Build a human-readable label → ID mapping
        options = {
            f"ID {r['id']} · {r['requester']} · ${r['amount']:,.2f}": r['id']
            for r in records
        }
        selected = st.selectbox('Select request', list(options.keys()))
        new_status = st.radio('New status',
                              ['Pending', 'Approved', 'Rejected'],
                              horizontal=True)
        if st.button('Update'):
            update_status(options[selected], new_status)
            st.success(f'Updated to {new_status}')
            st.rerun()   # refresh the page to show updated data
    else:
        st.info('No requests to update.')
