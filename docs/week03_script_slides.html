The Problem

🗣 🗣 Script

Imagine you have two Python projects. Project A needs requests version 2.28. Project B needs requests version 2.31. If you install packages globally — into your system Python — you can only have one version. Installing 2.31 for Project B breaks Project A. This is the dependency conflict problem, and it is the reason virtual environments exist.

The Private Room Mental Model

💻 💻 Draw on screen or whiteboard

Draw two boxes labelled 'Project A (.venv)' and 'Project B (.venv)'. Draw arrows showing each has its own packages. Draw a third box 'System Python' showing no packages from either project.

DO: Draw two boxes labelled 'Project A (.venv)' and 'Project B (.venv)'. Draw arrows showing each has its own packages. Draw a third box 'System Python' showing no packages from either project.
🗣 🗣 Script

A virtual environment is a private room for one project. The room has its own copy of Python and its own shelves of packages. When you activate the venv, you walk into the room. Every package you install goes onto that room's shelves. When you deactivate, you walk out. The packages you installed stay in the room. They do not spill into other projects. Other projects cannot touch them. This is isolation. This is what venvs give you.
