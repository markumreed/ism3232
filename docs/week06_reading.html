Conditionals — business rules as code

Business rules are conditionals

Every approval threshold, every routing rule, every eligibility check is an if/elif/else statement. The code structure mirrors the business policy.

Multi-tier approval routing
LIMIT_AUTO     =  500.00
LIMIT_MANAGER  = 2000.00
LIMIT_DIRECTOR = 5000.00

def get_approver(amount: float) -> str:
    if amount <= LIMIT_AUTO:
        return 'Auto-approved'
    elif amount <= LIMIT_MANAGER:
        return 'Manager'
    elif amount <= LIMIT_DIRECTOR:
        return 'Director'
    else:
        return 'Board'

# Test every boundary
for amt in [500, 501, 2000, 2001, 5000, 5001]:
    print(f'${amt:>6,.2f}  →  {get_approver(amt)}')
⚠ Test boundary values

Testing with 100 and 1000 is not enough. Test exactly at the threshold (500) and one unit above (501). The boundary is where bugs hide.

Lists and the accumulator pattern

List operations and the accumulator
amounts = [200.00, 1500.00, 400.00, 2200.00, 850.00]

# The accumulator pattern — initialise BEFORE the loop
total   = 0.0    # ← OUTSIDE the loop
flagged = []     # ← OUTSIDE the loop

for amount in amounts:
    total += amount              # add to running total
    if amount > 1000:
        flagged.append(amount)   # collect matching items

print(f'Total:   ${total:,.2f}')       # $5,150.00
print(f'Flagged: {flagged}')            # [1500.0, 2200.0]
⚠ Initialise outside the loop
⚖ Accumulator inside the loop — resets every iteration
✗ total always equals only the LAST amount
amounts = [200, 1500, 400]
for amount in amounts:
    total = 0          # resets to 0 on every pass
    total += amount    # = 200, then 1500, then 400
print(total)           # 400 — wrong
✓ Initialise before the loop — once
amounts = [200, 1500, 400]
total = 0              # once, before the loop
for amount in amounts:
    total += amount    # += 200, +=1500, +=400
print(total)           # 2100 — correct

The accumulator lives outside the loop. Inside the loop you only update it.

If you write total = 0 inside the loop body, it resets to zero on every iteration. The accumulator must be created before the loop starts.

Dictionaries — one record per dict, many records per list

Dict operations and the list-of-dicts pattern
# One record = one dict
request = {
    'id':        101,
    'requester': 'Taylor',
    'category':  'Equipment',
    'amount':    1250.00,
    'status':    'Pending',
}

print(request['requester'])  # Taylor
request['status'] = 'Approved'  # update a field

# Many records = list of dicts (one dict = one row)
records = [
    {'id': 101, 'requester': 'Taylor',  'amount': 1250.00, 'status': 'Pending'},
    {'id': 102, 'requester': 'Jordan',  'amount':  350.00, 'status': 'Pending'},
    {'id': 103, 'requester': 'Morgan',  'amount': 2800.00, 'status': 'Approved'},
]

# Filter: all Pending records
pending = [r for r in records if r['status'] == 'Pending']
print(f'Pending: {len(pending)} records')

# Total pending amount
total = sum(r['amount'] for r in pending)
print(f'Total pending: ${total:,.2f}')
⚖ == comparison instead of = assignment
✗ Nothing changes — this is a comparison, not an update
record['status'] == 'Approved'  # evaluates True/False, then discards
print(record['status'])          # still 'Pending'
✓ = for assignment
record['status'] = 'Approved'   # updates the dict
print(record['status'])           # 'Approved'

Single = assigns. Double == compares. They look similar but do completely different things.

The list-of-dicts table

One dict = one row. A list of dicts = the table. The dict keys are the column names. In Week 14 we replace this list with a real SQLite database — the structure and the operations are identical.

Key terms this week

if / elif / elseBranching statement — runs different code depending on which condition is True.
for loopIterates over every item in a sequence, executing the loop body once per item.
accumulator patternInitialise a variable before the loop; update it inside. Produces a running total or collected list.
append()Adds one item to the end of a list.
dict key accessrecord['key'] — retrieves the value for that key.
list-of-dictsOne dict per record; the list holds all records. One dict = one row, list = the table.

Why conditionals are the foundation of business logic

Every rule a business follows — "orders over $500 need manager approval," "employees with fewer than 90 days tenure cannot take vacation," "purchases from restricted vendors require two signatures" — is an if statement waiting to be written. The person who can translate a business policy document into working, tested conditionals is doing the actual job of a business application developer.

Notice how the real business rule already tells you what to write:

From policy to code — direct translation
# Policy: "Requests over $1,000 require manager approval.
#          Requests over $5,000 require director approval."

MANAGER_LIMIT  = 1_000.00
DIRECTOR_LIMIT = 5_000.00

def get_approver(amount: float) -> str:
    if amount <= MANAGER_LIMIT:    # not over $1,000 → auto
        return 'Auto'
    elif amount <= DIRECTOR_LIMIT:  # over $1,000 but not over $5,000
        return 'Manager'
    else:                           # over $5,000
        return 'Director'

The elif chain is ordered from least to most expensive. Each condition only needs to handle its range because the earlier conditions already ruled out smaller amounts. This ordering is intentional and important — reversing it breaks the logic silently.

Q1 · Multiple Choice
What does the accumulator pattern do?
  • a) Deletes items from a list one by one
  • b) Builds up a result across loop iterations using += or .append()
  • c) Counts backward from n to 0
  • d) Exits a loop when a condition is met
Answer
An accumulator starts at an empty/zero value, then grows with each iteration.
Q2 · Code Trace
What is printed?
amounts = [100, 250, 75, 400]
total = 0
for a in amounts:
    total += a
print(total)
Answer
825 — The accumulator adds each element: 0+100+250+75+400 = 825.
Q3 · Multiple Choice
Which expression accesses the value for key "status" in dict d?
  • a) d["status"]
  • b) d.status
  • c) d->status
  • d) d(status)
Answer
Square bracket notation is the standard way to access dict values by key.
Q4 · Code Trace
What is the output?
record = {"name": "Apex Laptop", "price": 899.00, "qty": 5}
print(record["name"])
print(record["price"] * record["qty"])
Answer
Apex Laptop 4495.0 — The first line accesses the "name" key; the second multiplies the two numeric values.
Q5 · Multiple Choice
What happens if you access a dict key that doesn't exist using square brackets?
  • a) Returns None
  • b) Returns an empty string
  • c) Raises a KeyError
  • d) Creates the key with value None
Answer
d["missing_key"] raises KeyError immediately. Use d.get("key") to get None instead of an error.
Q6 · Multiple Choice
In a list-of-dicts, what does each item in the list represent?
  • a) A column name
  • b) One record (row) of data
  • c) A table schema
  • d) A database connection
Answer
This is the standard pattern for tabular data: one list holding many records, each record a dict with the same keys.
Q7 · True / False
An elif branch runs even if the preceding if condition was True.
Answer
False — elif is skipped entirely if the preceding if (or elif) was True. Only the first matching branch executes.
Q8 · True / False
list.append() modifies the list in place and returns None.
Answer
True — append() modifies the list in place. It does not return the modified list — it returns None. A common mistake: my_list = my_list.append(x) sets my_list to None.
Q9 · Short Answer
What is the difference between a while loop and a for loop, and when would you choose each?
Answer
A for loop iterates over a known sequence a fixed number of times. A while loop repeats as long as a condition is True, with no predetermined count. Use for when you know the iterable; use while when waiting for a condition to change (user input, retry logic, draining a queue).
