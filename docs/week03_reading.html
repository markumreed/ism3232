Why virtual environments exist

💻 Windows users — WSL required

All commands on this page use zsh on Mac/Linux. Windows users must run these inside WSL (Windows Subsystem for Linux), not PowerShell and not Command Prompt. Open VS Code, press Ctrl+`, and confirm the terminal shows a Linux prompt (ending in %). If you have not installed WSL yet, complete the pre-course setup tutorial before continuing.

Imagine two projects. Project A needs requests==2.28. Project B needs requests==2.31. If you install packages globally — into your system Python — you can only have one version. Installing 2.31 breaks Project A. This is the dependency conflict problem.

The private room mental model

A virtual environment is a private room for one project. The room has its own copy of Python and its own packages. Activate the venv → you walk into the room. Deactivate → you walk out. Packages in one room do not affect any other project.

The complete venv workflow

Five steps — run in your module folder
# Step 1: Create the venv
python3 -m venv .venv

# Step 2: Activate it
source .venv/bin/activate
# Prompt now shows (.venv) — this confirms activation

# Step 3: Verify
which python3   # must show .venv/bin/python3
pip list        # minimal packages — only pip and setuptools

# Step 4: Install packages
pip install pytest ruff
pip list        # now shows pytest and ruff

# Step 5: Freeze to requirements.txt
pip freeze > requirements.txt
cat requirements.txt   # pinned versions
⚠ Never commit .venv/

Create a .gitignore file with .venv/ before your first git add. The venv folder is platform-specific and can be 50–100 MB. Anyone who clones your repo recreates it with pip install -r requirements.txt.

Create .gitignore before any git commands
echo '.venv/' > .gitignore
echo '__pycache__/' >> .gitignore
cat .gitignore
Recreating a venv from requirements.txt
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

Customising zsh with .zshrc

.zshrc is a hidden file in your home directory (~/.zshrc). zsh reads it every time a new terminal session opens. Aliases defined here are available in every future session automatically.

Open .zshrc in VS Code
code ~/.zshrc
Required aliases — add these below any existing content
# ISM3232 required aliases
alias ll='ls -la'
alias c='clear'
alias py='python3'
alias gs='git status'
alias ga='git add .'
alias gcmsg='git commit -m'
alias gp='git push'
alias gl='git log --oneline'
alias tree2='tree -L 2'

# Shell function
mkcd() { mkdir -p "$1" && cd "$1"; }
Apply changes to the current session
source ~/.zshrc

# Test — these should now work
ll              # runs ls -la
gs              # runs git status
mkcd testdir    # creates testdir/ AND enters it
pwd             # should end in /testdir
⚠ source is required

Aliases only become active after source ~/.zshrc. In future sessions they activate automatically when the terminal opens. Only in the current session do you need to source manually after editing.

Key terms this week

venvVirtual environment — an isolated Python + packages installation for one project.
source .venv/bin/activateActivates the venv. The (.venv) prefix confirms it.
which python3Shows which Python is currently active. Must show .venv/bin/python3 when activated.
pip freezeLists all installed packages with exact version numbers.
requirements.txtA text file listing pinned package versions so others can recreate the environment.
.gitignoreA file telling Git which files/folders to never track. .venv/ must be listed here.
.zshrczsh startup script. Aliases and functions defined here are always available.
aliasA shortcut that expands to a longer command when typed.
