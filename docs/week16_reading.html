The six required GenAI controls

Missing any single control means the GenAI requirement is not met. All six must be present and demonstrable at the final presentation.

  • 1. Disclosure label — AI output is always shown inside a labelled st.info() box.
  • 2. Human review gate — a second button ('Looks good — save') is required before the AI text is written to the database.
  • 3. No PII in prompts — only category and description are sent. No names, IDs, or amounts.
  • 4. Mocked tests — all tests in test_ai_feature.py use unittest.mock.patch. The real API is never called during testing.
  • 5. Limitations in README — at least two specific limitations listed.
  • 6. Student can explain — you must walk through every line at the presentation.
ai_feature.py — the minimal correct implementation
import anthropic

def summarise_request(category: str, description: str) -> str:
    """
    Return a one-sentence summary of a purchase request.
    Privacy: only category and description are sent.
    No names, IDs, or amounts are included.
    """
    client = anthropic.Anthropic()
    message = client.messages.create(
        model='claude-haiku-4-5-20251001',
        max_tokens=150,
        messages=[{
            'role': 'user',
            'content': (
                f'Summarise this {category} purchase request '
                f'in one sentence:\n\n{description}'
            )
        }]
    )
    return message.content[0].text.strip()
In app.py — disclosure label and human review gate
# Button 1: generate
if st.button('Generate AI Summary'):
    if description:
        summary = summarise_request(category, description)
        # Control 1: disclosure
        st.info(f'AI-generated summary (review before saving):\n\n{summary}')
        st.session_state['pending_summary'] = summary

# Control 2: human review gate
if 'pending_summary' in st.session_state:
    if st.button('Looks good — save this summary'):
        # save to database
        del st.session_state['pending_summary']
        st.success('Summary saved.')
        st.rerun()
tests/test_ai_feature.py — mocked tests (Control 4)
from unittest.mock import MagicMock, patch
from ai_feature import summarise_request

def test_summarise_returns_text():
    fake_content = MagicMock()
    fake_content.text = 'This is a one-sentence summary.'
    fake_message = MagicMock()
    fake_message.content = [fake_content]

    with patch('ai_feature.anthropic.Anthropic') as MockClient:
        MockClient.return_value.messages.create.return_value = fake_message
        result = summarise_request('Equipment', 'New standing desks.')

    assert result == 'This is a one-sentence summary.'

Module Quiz

Q1 · Multiple Choice
What is the primary privacy concern when passing data to an external AI API?
  • a) The API might be slow
  • b) Sensitive or identifying data may be stored or logged by the provider
  • c) The API might return incorrect JSON
  • d) The API call increases the database file size
Answer
ISM3232 requires sending only non-identifying fields (category, description) to the API, never names, IDs, or financial totals.
Q2 · Multiple Choice
What does the ISM3232 AI disclosure control require?
  • a) A pop-up warning before the page loads
  • b) A visible label or banner whenever AI-generated content is displayed
  • c) A log file recording every API call
  • d) An admin approval for each AI response
Answer
Users must always know when they are reading AI-generated text. Use st.info() or an explicit 'AI-generated summary' label.
Q3 · Multiple Choice
What is mocking in the context of testing an AI feature?
  • a) Writing fake test data for the database
  • b) Replacing the real API call with a controlled fake that returns a predictable response
  • c) Running the AI in offline mode
  • d) Testing only the UI, not the function
Answer
unittest.mock.patch replaces the Anthropic client during tests so tests make no real API calls, incur no cost, and do not fail due to network issues.
Q4 · Code Trace
What does this test verify?
def test_summarise_returns_text():
    fake = MagicMock()
    fake.text = 'A one-sentence summary.'
    with patch('ai_feature.anthropic.Anthropic') as mock_client:
        mock_client.return_value.messages.create.return_value.content = [fake]
        result = summarise_request('Equipment', 'Laptop for dev work')
    assert isinstance(result, str)
Answer
That summarise_request() returns a string — The test mocks out the Anthropic client (no real API call), then asserts the return type is str. It does not test the content, only the return type.
Q5 · Multiple Choice
Which Anthropic API parameter caps the length of the AI response?
  • a) temperature
  • b) max_tokens
  • c) model
  • d) top_p
Answer
max_tokens limits how many tokens the model generates. Setting it low (e.g., 100) for a one-sentence summary prevents runaway responses and controls cost.
Q6 · Multiple Choice
What is the required input control before sending user data to the AI?
  • a) Strip whitespace only
  • b) Validate and limit input: check non-empty, cap length, allow only expected values
  • c) Encrypt the input
  • d) Convert all text to uppercase
Answer
Control 2 of the six required controls. Always validate before the API call: reject empty input, enforce length limits, restrict to expected categories.
Q7 · True / False
The Anthropic client should be created inside the function that calls the API, not at the module level.
Answer
True (by ISM3232 convention) — Creating the client inside the function makes it easier to mock in tests. A module-level client requires more complex patching to replace during testing.
Q8 · True / False
AI-generated output may be displayed to users without any label or disclosure.
Answer
False — ISM3232 requires an AI disclosure label whenever AI-generated content is shown. Users must always know what was written by a human and what was generated by AI.
Q9 · Short Answer
What are the six required AI controls in the ISM3232 GenAI feature, and why does each matter?
Answer
(1) Disclosure — users see a label on AI output (honesty); (2) Input validation — reject empty or too-long input before the call (safety); (3) Input scoping — send only non-identifying fields (privacy); (4) Output guardrail — validate or trim AI output before display (reliability); (5) max_tokens limit — prevent runaway responses (cost and UX); (6) Error handling — show a user-friendly message on API failure, not a traceback (resilience).
