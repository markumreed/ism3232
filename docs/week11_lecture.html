Section 1 — Composition: Classes That Contain Other Classes
Composition: the manager class pattern

In most business systems, one object holds a collection of other objects. A RequestManager holds many BusinessRequest instances. This is composition — building complex objects from simpler ones by having one class contain instances of another.

Composition is more common than inheritance in business applications. Prefer composition when you need to manage or aggregate a collection of business entities.

RequestManager — a class that manages a collection of BusinessRequest objects
class RequestManager:
    """Manages a collection of business requests."""

    def __init__(self):
        self.requests = []   # starts empty — instances added later

    def add_request(self, request):
        """Add a BusinessRequest instance to the manager."""
        self.requests.append(request)

    def list_pending(self):
        """Return a list of all requests with status Pending."""
        return [r for r in self.requests if r.status == "Pending"]

    def total_amount(self):
        """Return the total value of all requests."""
        return sum(r.amount for r in self.requests)

    def get_by_status(self, status):
        """Return all requests matching the given status."""
        return [r for r in self.requests if r.status == status]

    def summary_report(self):
        """Print a formatted summary of all requests by status."""
        pending  = self.list_pending()
        approved = self.get_by_status("Approved")
        print(f"Total: {len(self.requests)} requests | ${self.total_amount():,.2f}")
        print(f"Pending: {len(pending)} | Approved: {len(approved)}")
Using composition — manager + requests working together
from models import BusinessRequest, RequestManager

manager = RequestManager()
manager.add_request(BusinessRequest(101, "Taylor", "Travel", 1200))
manager.add_request(BusinessRequest(102, "Jordan", "Equipment", 450))
manager.add_request(BusinessRequest(103, "Morgan", "Software", 3500))

manager.requests[0].approve()   # approve request 101 directly

manager.summary_report()
pending = manager.list_pending()
for r in pending:
    print(f"  {r.requester}: ${r.amount:,.2f} — review: {r.requires_review()}")
Section 2 — Inheritance: When to Use It
Inheritance in business applications: use sparingly

Inheritance lets one class inherit attributes and methods from a parent class. In business systems it is useful for modelling hierarchical relationships — a TravelRequest and an EquipmentRequest are both types of BusinessRequest, sharing core attributes but adding their own.

Do not use inheritance simply to avoid repeating code. Use it when there is a genuine is-a relationship: a TravelRequest is a BusinessRequest, not just something that has similar attributes.

Light inheritance — specialised request types
class TravelRequest(BusinessRequest):  # inherits from BusinessRequest
    """A business request specifically for travel."""

    def __init__(self, request_id, requester, amount, destination):
        super().__init__(request_id, requester, "Travel", amount)
        self.destination = destination   # extra attribute

    def requires_review(self):
        """Travel requests over $500 require review (lower threshold)."""
        return self.amount > 500   # overrides parent method

trip = TravelRequest(201, "Morgan", 850, "Atlanta")
print(trip.destination)        # Atlanta
print(trip.requires_review())  # True (850 > 500, travel threshold)
trip.approve()                 # inherited from BusinessRequest
print(trip.status)             # Approved
