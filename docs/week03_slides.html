📌 Mental model for students

A venv is like a private room for each project. The room has its own Python and its own shelves of packages. When you activate it, you walk in. When you deactivate, you walk out. Other rooms — other projects — are completely unaffected.
“Without a virtual environment, every Python package you install goes into one shared global installation. This works fine for one project. It breaks when Project A needs requests==2.28 and Project B needs requests==2.31 . They cannot both be installed globally at once. One of them breaks.”

⚠ Never commit .venv to GitHub

The .venv/ folder is large, platform-specific, and unnecessary — requirements.txt is what you share. Add .venv/ to your .gitignore before your first push. We will cover .gitignore in Week 4 but start the habit now.

Why Virtual Environments?

0–15 min  Why venv? Concept

🧩 The problem venv solves

Without a virtual environment, every Python package you install goes into one global installation shared by every project. This works until Project A needs requests==2.28 and Project B needs requests==2.31 — they conflict. A virtual environment gives each project its own private…
“In this course: every assignment uses a venv. The pre-submission ritual requires it to be active before you run ruff or pytest. Starting now, this is non-negotiable.”

🖥 Terminal

# 1. Create the virtual environment
#    Always name it .venv (the dot hides it from normal ls)
python3 -m venv .venv

# 2. Activate it
#    Your prompt changes to show (.venv) — this confirms it is active
source .venv/bin/activate

# 3. Confirm you are inside the venv
which python3        # should point inside .venv/bin/
pip list             # shows minimal packages

# 4. Install required tools for every project in this course
pip install pytest ruff

# 5. Freeze the environment to requirements.txt
pip freeze > requirements.txt
cat requirements.txt     # verify it was written correctly

# 6. To recreate the environment on another machine
pip install -r requirements.txt

# 7. Deactivate when done
deactivate

🧩 What is .zshrc?

.zshrc is a hidden configuration file in your home directory that zsh reads every time a new terminal opens. It is a startup script. Anything defined inside it — aliases, functions, environment variables — is automatically available in every terminal session. Open it with code…
“ .zshrc is a hidden configuration file in your home directory. zsh reads it every time a new terminal opens. Think of it as a startup script. Anything defined inside it — aliases, functions — is automatically available in every session.”

🖥 Terminal

# Open the config file
code ~/.zshrc

# --- Paste these lines into .zshrc ---

# Navigation shortcuts
alias ll='ls -la'           # detailed listing including hidden files
alias c='clear'             # clear the screen
alias tree2='tree -L 2'     # 2-level directory tree

# Python shortcut
alias py='python3'          # type py instead of python3

# Git shortcuts (used in the pre-submission ritual)
alias gs='git status'
alias ga='git add .'
alias gcmsg='git commit -m'
alias gp='git push'
alias gl='git log --oneline'

# Shell function: create a folder and immediately enter it
mkcd() { mkdir -p "$1" && cd "$1" }
