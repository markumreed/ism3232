⚖ Class-level list — all managers share the same list
✗ Adding to mgr1 contaminates mgr2
class RequestManager:
    requests = []   # class-level — ONE list for ALL instances

mgr1 = RequestManager()
mgr2 = RequestManager()
mgr1.add_request(req)
print(len(mgr2.requests))   # 1 — mgr2 is contaminated!
✓ Instance-level list — each manager gets its own
class RequestManager:
    def __init__(self):
        self.requests = []   # fresh list for each instance

mgr1 = RequestManager()
mgr2 = RequestManager()
mgr1.add_request(req)
print(len(mgr2.requests))   # 0 — independent

Any mutable object (list, dict) defined at class level is shared across all instances. Always initialise it inside __init__.

Composition — the manager pattern

has-a relationship

Composition means one class holds a collection of another class. RequestManager HAS A list of BusinessRequest objects. The manager coordinates; the entity owns its own methods.

RequestManager — all five required methods
class RequestManager:
    """Manages a collection of BusinessRequest objects."""

    def __init__(self) -> None:
        self.requests = []   # ← instance-level, NOT class-level

    def add_request(self, request) -> None:
        self.requests.append(request)

    def list_by_status(self, status: str) -> list:
        return [r for r in self.requests if r.status == status]

    def total_amount(self) -> float:
        return sum(r.amount for r in self.requests)

    def summary_report(self) -> None:
        pending  = self.list_by_status('Pending')
        approved = self.list_by_status('Approved')
        print(f'Total:    {len(self.requests)} requests')
        print(f'Pending:  {len(pending)}')
        print(f'Approved: {len(approved)}')
        print(f'Value:    ${self.total_amount():,.2f}')
✍ Think before reading on

If requests = [] is written at class level (outside __init__), and you create two managers, then add a request to manager 1 — how many requests will manager 2 show? Write your prediction before seeing the demo.

⚠ The class-level list trap

Writing requests = [] outside __init__ makes it class-level — shared by ALL instances. Every RequestManager would see every other manager's requests. Always write self.requests = [] inside __init__.

Light inheritance — the is-a relationship

TravelRequest inherits BusinessRequest
class TravelRequest(BusinessRequest):
    """A travel request with a lower approval threshold."""

    TRAVEL_LIMIT = 500.00

    def __init__(self, req_id, requester, amount, destination):
        super().__init__(req_id, requester, 'Travel', amount)
        self.destination = destination

    def requires_review(self) -> bool:
        return self.amount > self.TRAVEL_LIMIT

is-a test: 'A TravelRequest IS A BusinessRequest.' If that sentence makes sense, inheritance is appropriate. Use super().__init__() to call the parent constructor before adding child-specific attributes.

Why composition exists — and when to choose it over inheritance

The question every student asks in Week 11: when do I use composition and when do I use inheritance? The answer is a single test you can apply to any pair of classes.

The is-a / has-a test

Inheritance (is-a): "A TravelRequest IS A BusinessRequest." If this sentence makes grammatical and logical sense — if the child genuinely is a more specific version of the parent — use inheritance.

Composition (has-a): "A RequestManager HAS A list of BusinessRequests." If one class owns or manages a collection of another — use composition.

The mistake students make is reaching for inheritance when they want to reuse code. Inheritance is not about code reuse — it is about expressing a genuine type relationship. If you cannot say "A X IS A Y" with a straight face, use composition instead.

Testing the is-a sentence
# These sentences work — inheritance is appropriate:
# "A TravelRequest IS A BusinessRequest."       ✓
# "A SavingsAccount IS A BankAccount."          ✓
# "A Manager IS AN Employee."                   ✓

# These sentences do not work — use composition:
# "A RequestManager IS A BusinessRequest."      ✗ (manages them, not IS one)
# "A ShoppingCart IS A Product."                ✗ (contains them, not IS one)
# "A Department IS AN Employee."                ✗ (has employees, not IS one)
