OOP checkpoint — this is the last gate before the capstone
Students who cannot build a two-class system with passing tests by the end of Week 12 will struggle with every layer of the capstone. Use this week to identify and address gaps before SQL and Streamlit are introduced on top of OOP.
Section 1 — Design First — Before Any Code
1

Design First — Before Any Code 0–15 min

Lecture · Students do not open VS Code yet

🗣 Say — the design habit
“This week you design and build your own multi-class system from scratch. Before you type a single line of code, you are going to write your class design in comments. A class design written in five minutes saves an hour of refactoring.”
✏ Design template to write on board
# Entity class 1: [Name] | Attributes: [...] | Methods: [...]  ·  Entity class 2: [Name] | Attributes: [...] | Methods: [...]  ·  Manager class: [Name] | Holds: list of [entity] | Methods: add, list_by_status, total, report  ·  Business rules (will become methods and tests): [...]
🗣 Say — the constraint
“You cannot start coding until you have written the design. Show me your design before I will answer any code questions. If you cannot write the design, you do not yet know what you are building.”
💡 Suggested domains if students are stuck
Purchase request tracker · Leave request system · IT ticket tracker · Inventory alert manager · Employee onboarding checklist. Any domain with discrete records, clear business rules, and a status field works.
Section 2 — Build the Multi-Class System
2

Build the Multi-Class System 15–40 min

Independent work · Instructor circulates + reviews designs

🗣 Say — instructions
“Build your system now. Two entity classes, one manager class. Every class needs a docstring. Every business logic method needs a return value, not a print(). I will circulate and review designs first, then code.”
💡 What to look for while circulating
Classes that only have __init__ and no methods — push students to add at least two business logic methods. Methods that print instead of return — redirect to return values. Students who are writing the manager class first — suggest starting with the entity class since the manager depends on it.
⚠
Do not let students start with copy-pasting the example
The Employee/Department example in the lecture page is a reference, not a template. Students must design their own domain. Copied code cannot be explained in the capstone presentation.
Section 3 — Write 5+ pytest Tests
3

Write 5+ pytest Tests 40–60 min

Guided · Show the test structure once, then students write their own

🗣 Say — the test requirement
“Five tests minimum. At least one must test a boundary or edge case — where the business rule changes. If your approval threshold is $1,000, test with exactly $1,000.”
Test structure reminder — show once
from models import YourEntityClass, YourManagerClass

def test_your_method_basic():
    obj = YourEntityClass(...)
    assert obj.your_method() == expected_value

def test_boundary_case():
    obj = YourEntityClass(..., amount=1000)
    assert obj.requires_review() is False  # exactly at limit
💬 Ask after tests are written
“Can you explain in plain English what each test is checking and why that test matters for the business?” If a student cannot explain a test, they did not write it intentionally.
