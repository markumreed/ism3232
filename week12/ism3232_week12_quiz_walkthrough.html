How to use this walkthrough
Week 12 is the OOP checkpoint before the capstone. This walkthrough emphasises design thinking and testing rigour. Q5 (the class-level list bug) and Q8 (self.status vs status) are the two most common implementation bugs in student capstone code. Q11 (deactivate test) is an example of the test pattern every student must be able to write for their own capstone classes.
Question-by-question walkthrough
1
Multiple Choice — Auto-graded
Why is it important to write a class design before writing code?
A.Python requires it○
⚠ Python does not require a design document — it is a pedagogical requirement for this course.
B.It makes the code run faster○
⚠ Design does not affect runtime performance.
C.Designing first forces you to think through what the class needs before writing syntax — preventing structural mistakes✓
D.It is required by ruff○
⚠ ruff checks Python syntax, not design documents.
💬 Discussion — why C is correct

Writing the design first surfaces questions (what attributes? what should each method return?) before the cognitive load of syntax is added.

2
Multiple Choice — Auto-graded
What does "Out of Scope" mean in a design document?
A.Features you want to add later○
⚠ Features you want to add later are a backlog — Out of Scope means explicitly excluded from this version.
B.Features you will never build○
⚠ 'Features you will never build' is too permanent. Out of Scope is for this deliverable, not forever.
C.What you are explicitly NOT building in this version✓
D.The maximum number of records○
⚠ The maximum number of records is a capacity constraint, not scope definition.
💬 Discussion — why C is correct

Out of Scope lists what you are explicitly NOT building. It sets clear boundaries, prevents scope creep, and keeps the deliverable achievable.

3
Multiple Choice — Auto-graded
A boundary test for a method with threshold 1000 would test which values?
A.100 and 10000○
⚠ 100 and 10000 test the rule works in obvious cases but miss the exact edge where the rule changes.
B.999, 1000, and 1001✓
C.500 and 2000○
⚠ 500 and 2000 are reasonable but miss the exact values adjacent to the threshold.
D.Any two values above 1000○
⚠ Values above 1000 all pass — testing only above misses the critical near-boundary cases.
💬 Discussion — why B is correct

Boundary tests check the exact edge cases: the value just below (999), at (1000), and just above (1001) the threshold. These are where off-by-one bugs hide.

4
Multiple Choice — Auto-graded
Why should methods return values instead of printing them directly?
A.print() is slower○
⚠ print() being slower is not a meaningful distinction for business applications.
B.Methods that return values can be tested with assert; methods that only print cannot✓
C.Python does not allow print() in methods○
⚠ Python explicitly allows print() inside methods — it is a design choice, not a rule.
D.It is a style preference only○
⚠ Testability is a correctness concern, not style — it determines whether automated verification is possible.
💬 Discussion — why B is correct

assert result == expected tests the return value. If a method only prints, the test cannot capture or assert anything about the output.

5
Multiple Choice — Auto-graded
What is the most common OOP implementation bug that causes the manager independence test to fail?
A.Missing self on attributes○
⚠ Missing self on attributes is a different bug (produces AttributeError or creates local variable). The independence bug is specifically about where requests = [] is defined.
B.Using a class-level list (shared across instances) instead of an instance-level list (created in __init__)✓
C.Wrong method name○
⚠ Wrong method name would cause AttributeError when called, not a test failure about independence.
D.Missing __repr__○
⚠ Missing __repr__ would cause print() to show a memory address but would not affect data independence.
💬 Discussion — why B is correct

If self.requests is defined at the class level (outside __init__), all instances share the same list. It must be created inside __init__ as self.requests = [].

6
Multiple Choice — Auto-graded
A test is "trivial" when it asserts what?
A.A boundary condition○
⚠ Boundary conditions are valuable — trivial tests check obvious things that are always True.
B.An obvious constant like assert 1 == 1 or assert True✓
C.A method's return value○
⚠ Testing method return values is the GOAL — trivial tests assert something that never changes.
D.A default attribute value○
⚠ Testing default attribute values IS meaningful — it verifies the constructor works correctly.
💬 Discussion — why B is correct

Trivial tests pass regardless of whether your code is correct. Tests must assert your actual function's output against an expected value.

7
Multiple Choice — Auto-graded
What is wrong with: def approve(): inside a class body?
A.The method name is wrong○
⚠ approve is not a reserved keyword in Python.
B.approve is a reserved keyword○
⚠ The method name is valid and appropriate.
C.self is missing as the first parameter✓
D.The indentation is wrong○
⚠ The indentation is fine as long as the def is inside the class body.
💬 Discussion — why C is correct

Every instance method must have self as its first parameter. def approve(): will raise TypeError when called because Python cannot pass the instance.

8
Multiple Choice — Auto-graded
A student writes status = "Approved" inside an approve() method. What is wrong?
A.Nothing — this is correct○
⚠ Nothing is wrong is dangerous — this is a very common mistake that produces silent bugs.
B.It sets a local variable, not the instance attribute. Fix: self.status = "Approved"✓
C."Approved" should be in single quotes○
⚠ 'Approved' in single quotes is valid Python — quote style makes no difference here.
D.The method needs a return statement○
⚠ Status-changing methods typically return nothing (None) — that is acceptable design.
💬 Discussion — why B is correct

Without self., Python creates a local variable that exists only for the duration of the method call. The instance attribute is untouched.

Code Trace / Short Answer — Manual grading
What does this test check and what would cause it to fail?
def test_two_managers_are_independent(): mgr1 = RequestManager() mgr2 = RequestManager() mgr1.add_request(BusinessRequest(1, "A", "T", 500)) assert len(mgr2.requests) == 0
✓ Model Answer
It checks that adding to mgr1 does not affect mgr2. It fails if self.requests is a class-level list shared across instances instead of an instance-level list created in __init__.

Code Trace / Short Answer — Manual grading
What is wrong with this method?
def approve(self): status = "Approved"
✓ Model Answer
It sets a local variable called status, not the instance attribute self.status. The instance's status is unchanged after this method runs.
