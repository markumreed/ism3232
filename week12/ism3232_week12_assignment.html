Overview
This is your OOP checkpoint before the capstone. You must design and build a standalone multi-class business simulation — at least two entity classes and one manager class — for a domain different from Weeks 10 and 11. No database. No Streamlit. Pure OOP. This assignment confirms you can design and build a class system independently before SQL and Streamlit are layered on top in Unit 4.
Requirements
1
design.md committed before any class code

Write and commit design.md first. It must contain:

• Business domain (2–3 sentences, different from Weeks 10–11)

• Entity Class 1: name, all attributes with types, all methods with descriptions and return types

• Entity Class 2: name, all attributes, all methods

• Manager Class: name, what it holds, all methods

• Business rules that will become methods and test cases (at least 3)

• Out of scope: what you are explicitly not building

2
Two entity classes in models.py

Each entity class must have:

• __init__ with 4+ attributes, at least one with a default value

• At least 2 business logic methods with docstrings and return values

• __repr__

3
One manager class in models.py

The manager must have: add, list_by_status, total, and summary_report methods — all with docstrings. The manager holds instances of one or both entity classes.

4
main.py: a simulation that demonstrates everything

The simulation must: create a manager, add at least 4 entity instances of at least two different types, call status-changing methods, call business rule methods, call summary_report(), and print output that tells a coherent business story.

5
tests/test_models.py: at least seven tests

Required: default state test, each business logic method tested, at least one boundary test, manager independence test, manager filter test. All must pass.

⚠ Common mistakes
  • Using the same domain as Weeks 10/11 — the domain must be different
  • Committing models.py and design.md in the same commit — design must come first
  • Manager class missing summary_report() — this is a required method
  • Entity classes that are nearly identical — they should represent genuinely different business entities
