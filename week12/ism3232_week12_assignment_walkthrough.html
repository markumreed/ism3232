Requirement-by-Requirement Walkthrough
1
design.md committed before models.py

Check the Git commit history. design.md must have a commit timestamp earlier than models.py. This is verified by inspecting the repo commit log.

🔍 What the grader checks
design.md exists in the repository
design.md committed in an earlier commit than models.py (check commit timestamps in GitHub)
design.md has all required sections: domain, both entity classes with attributes/methods, manager class, business rules, out of scope
Domain is different from Weeks 10 and 11
💡 How to check commit order on GitHub
Go to the repository. Click 'Commits'. Find the commit that added design.md and the commit that added models.py. If they are in the same commit or models.py came first, the design-first requirement is not met.
⚠ Common mistakes at this step
design.md and models.py committed in the same commit — design-first not demonstrated; deduct 10/15
design.md committed after models.py — clear violation; deduct 15/15 for this requirement
Domain same as Week 10 or 11 — note it, deduct 5 pts
2
Two entity classes in models.py

Verify both classes meet the structural requirements independently.

🔍 What the grader checks
Two distinct entity classes (not slight variations of each other)
Each has __init__ with 4+ attributes and at least one default value
Each has at least 2 business logic methods with docstrings and return values
Each has __repr__
The two classes represent genuinely different business entities
⚠ Common mistakes at this step
Two classes but nearly identical logic — e.g. Employee and Contractor with same methods — partial credit (15/25)
One class has __repr__, the other does not — deduct 5 pts
Methods that print instead of return — deduct 5 pts per method
3
Manager class with all four methods

Same requirements as Week 11 — add, list_by_status, total, summary_report — all with docstrings.

🔍 What the grader checks
Manager class holds instances of at least one entity class
All four required methods present with docstrings
self.requests = [] inside __init__ (not class-level)
summary_report() produces formatted output
4
main.py simulation

The simulation must tell a coherent business story — not just create instances and print them.

🔍 What the grader checks
At least 4 entity instances of at least 2 different types
Status-changing methods called on multiple instances
Business logic methods called and results used in conditional logic (if entity.requires_review(): ...)
summary_report() called and output shows the correct state after all operations
5
Seven pytest tests with required coverage

Count tests and verify each required type is present.

🔍 What the grader checks
7+ test functions
Default state test for each entity class
Business logic method tested (bool True and False)
At least one boundary test (exact threshold value)
Manager independence test
Manager filter test
All 7 pass
Known Common Mistakes
⚠ Known common mistakes (from the assignment page)
Using the same domain as Weeks 10/11 — the domain must be different
Committing models.py and design.md in the same commit — design must come first
Manager class missing summary_report() — this is a required method
Entity classes that are nearly identical — they should represent genuinely different business entities

Grading Flow

Open design.md: check all required sections, verify domain is different from W10/W11.
Open models.py: audit both entity classes (4+ attrs, 2+ methods, __repr__) and manager class (4 methods, instance-level list).
Open main.py: verify 4+ instances of 2+ types, status changes called, report printed.
Open tests: count (7+), map to required types, check boundary test uses exact threshold.
Check ritual screenshot: 7+ tests green, ruff clean.

Rubric discussion
The design-first requirement (15 pts) is verifiable only through the commit history. Check it specifically — do not rely on the student's word. The simulation (15 pts) should be graded on coherence: does the code tell a story about a real business scenario, or is it just API calls with no narrative purpose? Full marks require both technical completeness and domain relevance.
