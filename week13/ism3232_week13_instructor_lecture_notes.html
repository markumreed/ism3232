Unit 4 begins — proposal requires instructor sign-off before Week 14
Do not let students begin building their capstone database or Streamlit app without an approved proposal. Students who skip the proposal and start coding produce systems that do not hold together. The 10-field proposal forces the design thinking that makes the build go smoothly.
Section 1 — Project Proposal — All 10 Fields
1

Project Proposal — All 10 Fields 0–20 min

Lecture · Walk through each field with examples

🗣 Say — framing
“The proposal forces you to think through the full design before a single line of capstone code is written. A student who cannot complete all 10 fields does not yet know what they are building — and that uncertainty will hurt every subsequent week.”
✏ Write the 10 fields on board
Project Name · Business Problem · Primary User · Records Stored · Business Rules (2+) · OOP Classes · SQL Tables · Streamlit Features · GenAI Feature · Out of Scope
🗣 Say — the Out of Scope field
“The Out of Scope field is as important as the rest. Students who do not constrain their scope build too much or too little. A clear out-of-scope statement keeps the capstone focused and achievable in four weeks.”
💡 Good capstone domains
Purchase request tracker · Leave request system · IT helpdesk ticket tracker · Inventory alert manager · Vendor bid comparison. Any domain with discrete records, clear business rules, and a status field (Pending / Approved / Rejected) works well.
⚠
Approve proposals before Week 14
Review proposals during or immediately after class. Students whose proposals are too vague, too large, or missing key fields need feedback before they begin building. A 10-minute proposal review conversation prevents a 4-week capstone failure.
Section 2 — SQL Fundamentals — Live Demo
2

SQL Fundamentals — Live Demo 20–45 min

All required SQL operations · Business context throughout

🗣 Say — framing
“SQL is how business data is stored and retrieved at scale. SQLite stores the entire database in a single .db file — no server required. The SQL you learn today applies directly to PostgreSQL, MySQL, and every production database.”
Required SQL operations — run each one live
-- Create a table (run once)
CREATE TABLE requests (
    id        INTEGER PRIMARY KEY,
    requester TEXT    NOT NULL,
    category  TEXT    NOT NULL,
    amount    REAL,
    status    TEXT    DEFAULT 'Pending',
    notes     TEXT
);

-- Insert a record
INSERT INTO requests (requester, category, amount)
VALUES ('Taylor', 'Travel', 1200.00);

-- Query all records
SELECT * FROM requests;

-- Filter records
SELECT * FROM requests WHERE status = 'Pending';

-- Sort records
SELECT * FROM requests ORDER BY amount DESC;

-- Update a record
UPDATE requests SET status = 'Approved' WHERE id = 1;

-- Aggregate report
SELECT status, COUNT(*), SUM(amount)
FROM requests GROUP BY status;
⚠
Parameterised queries — plant the seed now
Tell students: “When we connect Python to SQLite in Week 14, you will use ? placeholders instead of putting values directly in the SQL string. Building SQL strings with f-strings is a SQL injection vulnerability. We will cover this next week.”
Section 3 — sqlite3 Shell Exploration
3

sqlite3 Shell Exploration 45–60 min

Students run their own queries in the sqlite3 shell

sqlite3 shell commands — students run these
sqlite3 requests.db

.tables                  -- list all tables
.schema requests         -- show CREATE TABLE statement
.mode column             -- formatted column output
.headers on              -- show column names
SELECT * FROM requests;  -- query the table
.quit                    -- exit
🗣 Say — after exploration
“The sqlite3 shell is your database debugging tool. When something goes wrong in your Python code next week, open the shell and query the database directly. If the data is there, the problem is in Python. If it is not, the problem is in the SQL.”
Section 4 — Students Draft Their Own Schema
4

Students Draft Their Own Schema 60–70 min

Independent work · Instructor circulates

🗣 Say — instructions
“Using the domain from your proposal, write your CREATE TABLE statement now. Include all columns you identified in the proposal's Records Stored field. Add appropriate types and constraints.”
💡 What to check while circulating
Missing NOT NULL on required fields — point it out. Too many tables for the scope — suggest starting with one table and adding joins later if needed. Using spaces in column names — redirect to underscores. Missing a status column — almost every capstone domain needs one.
