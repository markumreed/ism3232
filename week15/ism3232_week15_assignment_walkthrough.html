Requirement-by-Requirement Walkthrough
1
Feature 1 — Add a record with validation

Check the Submit tab screenshot and the source code. Both validation and success/error message are required.

🔍 What the grader checks
Form fields present for all schema-required inputs
At least one validation check (if requester: or if amount > 0:)
add_record() from database.py called on valid submission
st.success() shown on success
st.error() shown on validation failure
Screenshot shows the success message after a submission
⚠ Common mistakes at this step
No validation — form submits empty records to the database
add_record() called with hardcoded values instead of widget values
st.write() used instead of st.success() — partial credit (12/20)
2
Feature 2 — View all records

Verify st.dataframe() is used and the empty-state is handled.

🔍 What the grader checks
get_all_records() called to retrieve data
st.dataframe(records, use_container_width=True) used
Empty state: if records: ... else: st.info('No records yet.')
⚠ Common mistakes at this step
st.table() used instead of st.dataframe() — partial credit (6/10)
No empty state handling — acceptable but note it
3
Feature 3 — Filter with count label

Test this by checking the screenshot: filter selectbox visible, count label below the table.

🔍 What the grader checks
At least one filter control (selectbox, text_input, or radio)
Filter applied to retrieved data in Python (not in SQL)
st.caption() or st.write() showing record count below the table
Screenshot shows filtered results (not all records when a filter is selected)
✅ Full credit filter
status_filter = st.selectbox('Filter', ['All', 'Pending', 'Approved']) records = get_all_records() if status_filter != 'All': records = [r for r in records if r['status'] == status_filter] st.dataframe(records) st.caption(f'{len(records)} record(s) shown')
❌ Always shows all (common bug)
status_filter = st.selectbox('Filter', ['Pending', 'Approved']) records = get_all_records() st.dataframe(records) # filter applied but records not updated
⚠ Common mistakes at this step
Filter selectbox present but records not re-filtered before display — the most common Feature 3 bug
Count label missing — partial credit (15/20)
Filter passed to SQL query instead of applied in Python — acceptable but note it
4
Feature 4 — Update status with live selectbox

The selectbox for record selection must be built from get_all_records() — not hardcoded IDs.

🔍 What the grader checks
get_all_records() called to populate the selectbox options
Selectbox shows meaningful labels (ID + requester name + amount)
update_status() called with the correct record_id and new_status
st.success() shown after update
Screenshot shows the success message after an update
💡 How to check the selectbox build
Look for: {r['id']: r['id'] for r in records} or options = {f"ID {r['id']}...": r['id'] for r in records}. If IDs are hardcoded, that is a violation.
⚠ Common mistakes at this step
Hardcoded record IDs in the selectbox — fails immediately when records are deleted or added; deduct 10 pts
update_status() not called — only a visual change with no database write; 0 pts
Change not reflected when View tab is opened after update — likely the update is not persisting
5
Feature 5 — Operational report

Verify the report uses get_status_report() (not get_all_records()) and shows count AND total per status.

🔍 What the grader checks
get_status_report() called (not get_all_records with filtering)
Output shows both count and total amount per status
st.dataframe() or st.metric() used (not plain st.write() with raw tuples)
Screenshot shows the report with real data and at least two different statuses
⚠ Common mistakes at this step
Report shows individual records instead of aggregates — wrong function used
Only count shown, not total amount — partial credit (10/15)
Report is always empty because get_status_report() is not wired to the same database

App configuration requirements

Check the source code for all four configuration requirements.

🔍 What the grader checks
st.set_page_config(page_title='...', layout='wide') at the top of the file
create_table() called outside any tab or button block (at the top level)
All features in separate tabs using st.tabs()
App title customised to the student's domain (not 'Request Tracker')
Known Common Mistakes
⚠ Known common mistakes (from the assignment page)
Hardcoded record IDs in the Update tab — the selectbox must be built from get_all_records()
Filter that always shows all records — test the filter by selecting a status that excludes some records
Report showing individual records instead of grouped aggregates — it must use get_status_report()
create_table() not called at the top of app.py — without it the app crashes on first run

Rubric discussion
The filter feature (20 pts) is the most commonly buggy. The selectbox is there but the filter is not applied to the records variable before display. Test this by inspecting the code: after the filter selectbox, does a line like 'records = [r for r in records if ...]' appear? If not, the filter does not work. The Update tab (20 pts) has a critical distinction: hardcoded IDs vs live data. Even one hardcoded ID loses 10 pts.
