Section 1 — Why Streamlit?
1

Why Streamlit? 0–15 min

Concept + first run live

🗣 Say — what Streamlit is
“Streamlit turns Python scripts into browser-based business tools without writing any HTML, CSS, or JavaScript. Every widget — text inputs, buttons, dropdowns, tables — is a Python function call. The interface re-runs the entire script top to bottom whenever the user interacts with it.”
✏ Write on board — the Streamlit execution model
User interacts → entire script re-runs top to bottom → UI updates. This is different from traditional web apps. Understand this model or your app will have confusing state bugs.
Install and first run
source .venv/bin/activate
pip install streamlit
pip freeze > requirements.txt

streamlit run app.py
# Browser opens at http://localhost:8501
🗣 Say — the rerun model consequence
“Because the entire script reruns on every interaction, anything you compute at the top of the script is recomputed every time. This is why database calls go inside tab blocks or conditional blocks — not at the top level.”
Section 2 — Build the Streamlit App Together
2

Build the Streamlit App Together 15–45 min

Live coding · Wire to database.py

🗣 Say before starting
“We are building the complete app skeleton together. Every widget I type, you type. We will add all five required features and wire each one to the database functions from Week 14.”
app.py skeleton — build together
import streamlit as st
from database import create_table, add_record, get_all_records, update_status, get_status_report

create_table()   # idempotent -- safe to call on every run

st.set_page_config(page_title='Request Tracker', layout='wide')
st.title('Business Request Tracker')

tab1, tab2, tab3 = st.tabs(['Submit', 'View & Filter', 'Report'])

with tab1:
    st.header('Submit a New Request')
    requester = st.text_input('Your name')
    category  = st.selectbox('Category', ['Travel', 'Equipment', 'Software'])
    amount    = st.number_input('Amount ($)', min_value=0.0, step=10.0)
    notes     = st.text_area('Notes (optional)')
    if st.button('Submit Request'):
        if requester and amount > 0:
            add_record(requester, category, amount, notes)
            st.success(f'Request submitted for {requester}')
        else:
            st.error('Name and amount are required.')

with tab2:
    records = get_all_records()
    if records:
        st.dataframe(records, use_container_width=True)
    else:
        st.info('No records yet.')

with tab3:
    report = get_status_report()
    st.dataframe(report)
💡 Status update feature — common implementation gap
The update status feature requires knowing the record ID. Show students how to get the ID from the records list, display it in a selectbox, and pass it to update_status(). This is the feature students most often implement incorrectly.

Section 3 — FastAPI — Concept Lecture Only
3

FastAPI — Concept Lecture Only 60–75 min

No deliverable · Concepts and one demo route

⚠
FastAPI is concept only — no assignment, no capstone requirement
There is no FastAPI deliverable. Students should understand what an API is, what routes and HTTP methods mean, and how FastAPI structures a Python web service. That is the learning objective. Do not spend more than 15 minutes here.
🗣 Say — what an API is
“An API is a contract between a client and a server. The client sends a request to a specific URL with a specific HTTP method. The server returns data, usually as JSON. This is how every web service in the world communicates.”
Minimal FastAPI demo — run with: uvicorn main:app --reload
from fastapi import FastAPI

app = FastAPI()

@app.get('/')
def home():
    return {'message': 'Business API is running'}

# Visit http://127.0.0.1:8000/docs for auto-generated documentation
🗣 Say — the connection to Streamlit
“Your Streamlit app is a client. If your business logic lived behind a FastAPI service, Streamlit would call the API instead of importing database.py directly. That is the architecture for production systems. For this course, we import directly — simpler and sufficient.”
