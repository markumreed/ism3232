Overview
Build the user-facing interface for your capstone. All five Streamlit features must be implemented and wired to your database.py functions from Week 14. Every feature must work end-to-end — submitting, viewing, filtering, updating, and reporting on real data stored in SQLite.
Requirements
1
Feature 1: Add a record

A form with input widgets for every required field in your schema. On submission:

• Validate that required fields are not empty

• Call add_record() from database.py

• Show a st.success() message on success or st.error() on validation failure

Screenshot required: Tab showing a successful submission with the success message visible.

2
Feature 2: View all records

Display all records from get_all_records() in a st.dataframe() with use_container_width=True. If there are no records, show st.info('No records yet.').

3
Feature 3: Filter or search records

Provide at least one filter control (a st.selectbox or st.text_input) that narrows the displayed records. The filter must operate on the retrieved data in Python — you do not need to pass it to the SQL query. Show a record count below the filtered table.

Screenshot required: Tab showing filtered results with the count label.

4
Feature 4: Update status

Allow the user to select a record by ID (using a selectbox built from live data) and change its status. Call update_status() after confirmation. Show a success message. When the View tab is opened after the update, the change must be reflected.

Screenshot required: Tab after a successful status update.

5
Feature 5: Operational report

Display the output of get_status_report() as either st.dataframe() or st.metric() widgets — showing count and total per status. The report must update dynamically as records are added and statuses change.

Screenshot required: Report tab with real data showing at least two different statuses.

6
App configuration and structure

• st.set_page_config(page_title='Your App Name', layout='wide') at the top

• create_table() called at the top of the script (idempotent — safe to call every run)

• Features organised into tabs using st.tabs()

• App title customised to your business domain (not “Request Tracker”)

⚠ Common mistakes
  • Hardcoded record IDs in the Update tab — the selectbox must be built from get_all_records()
  • Filter that always shows all records — test the filter by selecting a status that excludes some records
  • Report showing individual records instead of grouped aggregates — it must use get_status_report()
  • create_table() not called at the top of app.py — without it the app crashes on first run
