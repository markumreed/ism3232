Section 1 — What Is a Class? The Mental Model
1

What Is a Class? The Mental Model 0–15 min

Lecture only · Build the intuition before showing any code

🗣 Say — the blueprint analogy
“A class is a blueprint. An object is a specific thing built from that blueprint. The BusinessRequest class describes what every request has and what every request can do. Each individual request — request #101, request #102 — is a separate instance of that class.”
✏ Write on board — two columns
CLASS (blueprint): defines attributes · defines methods · written once · used to create many instances | OBJECT / INSTANCE (a specific thing): has its own data · can call any method · changing one does not affect others
🗣 Say — the business analogy
“The BusinessRequest class is the purchase request form template. req_101 is a filled-out form for Taylor's laptop request. req_102 is Jordan's equipment request. Same template, different data, completely independent.”
💡 The question that reveals understanding
Ask any student: “If I call req_101.approve(), does req_102.status change?” The correct answer is no. If they hesitate, the mental model is not there yet.
Section 2 — Writing the BusinessRequest Class
2

Writing the BusinessRequest Class 15–40 min

Live demo · Explain every line before typing it

🗣 Say before starting
“We are going to build this class together. I will explain what each line means before we type it. No copy-paste.”
BusinessRequest — build line by line
class BusinessRequest:
    """Represents a single business purchase or travel request."""

    def __init__(self, request_id, requester, category, amount):
        self.request_id = request_id
        self.requester  = requester
        self.category   = category
        self.amount     = amount
        self.status     = 'Pending'   # default

    def approve(self):
        self.status = 'Approved'

    def reject(self):
        self.status = 'Rejected'

    def requires_review(self):
        return self.amount > 1000

    def __repr__(self):
        return f'BusinessRequest({self.request_id}, {self.requester}, ${self.amount}, {self.status})'
⚠
Explain self on every method — do not skip this
Point to self in each method signature and say: “self is Python's way of saying 'the specific instance this method was called on.' When you call req_101.approve(), self is req_101. When you call req_102.approve(), self is req_102.”
Section 3 — Creating Instances and Calling Methods
3

Creating Instances and Calling Methods 40–55 min

Live demo · Show independence of instances

Create instances and call methods
req_101 = BusinessRequest(101, 'Taylor', 'Travel', 1200)
req_102 = BusinessRequest(102, 'Jordan', 'Equipment', 450)

print(req_101.requester)          # Taylor
print(req_101.status)             # Pending

print(req_101.requires_review())  # True
print(req_102.requires_review())  # False

req_101.approve()
print(req_101.status)             # Approved
print(req_102.status)             # Pending -- unchanged
💬 Key question to ask
“Why is req_102.status still Pending after we approved req_101?” Look for: because they are independent instances. Each has its own copy of every attribute. Approving one does not touch the other.
Section 4 — Encapsulation
4

Encapsulation 55–70 min

Concept · Link to maintainability

🗣 Say — what encapsulation means
“Encapsulation means bundling the data and the operations on that data into a single unit. The BusinessRequest class knows how to approve itself, reject itself, and determine whether it needs review. The caller does not need to know the internal logic — it just calls the method.”
🗣 Say — why it matters
“This is better design than checking req['amount'] > 1000 everywhere in your code. If the approval threshold changes, you update it in one place inside the class. Every caller gets the new behaviour automatically — the same reason we wrote functions in Week 7.”

AI use policy for OOP — state explicitly
