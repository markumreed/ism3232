Unit 3 begins — Object-Oriented System Design
OOP is the bridge between the Python fundamentals of Units 1–2 and the full-stack capstone of Unit 4. Students who understand classes and objects will be able to design their capstone data models. Students who do not will struggle with every layer on top of them.
Section 1 — What Is a Class? What Is an Object?
The core mental model

A class is a blueprint. An object is a specific thing built from that blueprint. A BusinessRequest class describes what every request has (attributes) and what every request can do (methods). Each individual request — request #101, request #102 — is a separate instance of that class.

Think of it this way: "BusinessRequest" is the concept. req_101 is an actual request with actual data. The class defines the shape; objects are the data.

Class = blueprint

Defines attributes every instance will have. Defines methods every instance can call. Written once. Used to create as many instances as needed.

Object / Instance = a specific thing

Created by calling the class like a function. Has its own copies of every attribute. Can call any method defined in the class.

Section 2 — Writing the BusinessRequest Class
BusinessRequest — write together line by line
class BusinessRequest:
    """Represents a single business purchase or travel request."""

    def __init__(self, request_id, requester, category, amount):
        self.request_id = request_id
        self.requester  = requester
        self.category   = category
        self.amount     = amount
        self.status     = "Pending"   # default — not passed in

    def approve(self):
        """Mark this request as approved."""
        self.status = "Approved"

    def reject(self):
        """Mark this request as rejected."""
        self.status = "Rejected"

    def requires_review(self):
        """Return True if the amount exceeds the approval limit."""
        return self.amount > 1000

    def __repr__(self):
        return f"BusinessRequest({self.request_id}, {self.requester}, ${self.amount}, {self.status})"
Section 3 — Creating and Using Instances
Creating instances and calling methods
# Create two separate instances from the same class
req_101 = BusinessRequest(101, "Taylor", "Travel", 1200)
req_102 = BusinessRequest(102, "Jordan", "Equipment", 450)

# Access attributes
print(req_101.requester)          # Taylor
print(req_101.status)             # Pending

# Call methods
print(req_101.requires_review())  # True (1200 > 1000)
print(req_102.requires_review())  # False (450 <= 1000)

req_101.approve()
print(req_101.status)             # Approved
print(req_102.status)             # Pending — unchanged

# Instances are independent — changing one does not affect the other
print(req_101)   # BusinessRequest(101, Taylor, $1200, Approved)
print(req_102)   # BusinessRequest(102, Jordan, $450, Pending)
Section 4 — Encapsulation
Encapsulation: data and behaviour belong together

Encapsulation means bundling the data (attributes) and the operations on that data (methods) into a single unit. The BusinessRequest class knows how to approve itself, reject itself, and determine whether it needs review. The caller does not need to know the internal logic — it just calls the method.

This is better design than checking req["amount"] > 1000 everywhere. If the threshold changes, you update it in one place inside the class.

Discussion: what goes inside a class?
What other methods might a BusinessRequest class need? Think about what a business process actually does to requests — routing, escalation, history tracking, notifications.
Look for: summary(), escalate(), add_note(), get_age_in_days(). Students start thinking about design, not just syntax.
