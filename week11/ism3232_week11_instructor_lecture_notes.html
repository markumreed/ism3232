Section 1 — Composition — The Manager Class Pattern
1

Composition — The Manager Class Pattern 0–20 min

Live demo · Build RequestManager

🗣 Say — framing
“In most business systems, one object holds a collection of other objects. A RequestManager holds many BusinessRequest instances. This is composition — building complex objects from simpler ones. It is more common in business applications than inheritance.”
Build RequestManager together
class RequestManager:
    """Manages a collection of business requests."""

    def __init__(self):
        self.requests = []   # starts empty

    def add_request(self, request):
        self.requests.append(request)

    def list_pending(self):
        return [r for r in self.requests if r.status == 'Pending']

    def total_amount(self):
        return sum(r.amount for r in self.requests)

    def get_by_status(self, status):
        return [r for r in self.requests if r.status == status]

    def summary_report(self):
        pending  = self.list_pending()
        approved = self.get_by_status('Approved')
        print(f'Total: {len(self.requests)} | ${self.total_amount():,.2f}')
        print(f'Pending: {len(pending)} | Approved: {len(approved)}')
🗣 Say — after writing
“Notice that RequestManager does not know how to approve a request — that is BusinessRequest's job. RequestManager knows how to hold, filter, and summarise requests. Each class has one clear responsibility.”

manager = RequestManager()

Section 3 — Light Inheritance
3

Light Inheritance 40–55 min

Brief · Show the is-a test · Do not over-teach

🗣 Say — when to use inheritance
“Use inheritance when there is a genuine is-a relationship. A TravelRequest is a BusinessRequest. An EquipmentRequest is a BusinessRequest. They share core attributes but add their own. Do not use inheritance just to avoid repeating code — that is what composition is for.”
Light inheritance example
class TravelRequest(BusinessRequest):
    def __init__(self, request_id, requester, amount, destination):
        super().__init__(request_id, requester, 'Travel', amount)
        self.destination = destination

    def requires_review(self):
        return self.amount > 500  # lower threshold for travel
⚠
Do not spend more than 15 minutes on inheritance
Composition is the pattern students will use in the capstone. Inheritance is context and occasional tooling. If the class runs long, cut the inheritance demo short and cover it in the assignment Q&A.

OOP to SQL Mapping 55–70 min

Critical for capstone design · Draw on board

🗣 Say — why this matters now
“In Week 13 you will design your capstone SQL schema. Students who understand that a class maps to a table, an attribute maps to a column, and an instance maps to a row will find SQL design intuitive. Students who do not will struggle to connect the two layers.”
✏ Draw the mapping on board
class ⇔ table  ·  instance ⇔ row  ·  attribute ⇔ column  ·  manager.requests ⇔ SELECT * FROM requests  ·  list_pending() ⇔ SELECT * WHERE status = 'Pending'

“Think about your capstone business domain. What is your main entity class? What attributes does it need? What methods represent real business operations on it?” This is pre-work for the Week 13 proposal.
