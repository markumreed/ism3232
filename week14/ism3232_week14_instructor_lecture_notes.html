Section 1 — The sqlite3 Connection Pattern
1

The sqlite3 Connection Pattern 0–15 min

Lecture + short demo · Memorise this pattern

🗣 Say — the pattern
“Python connects to SQLite through the sqlite3 module — built in, no installation required. The pattern is always the same: open a connection, create a cursor, execute SQL, fetch results, close. Using a context manager (with) handles the close automatically and commits on success.”
The connection pattern — type this together
import sqlite3

def get_connection():
    return sqlite3.connect('data/requests.db')

# Reading data
with get_connection() as conn:
    cursor = conn.cursor()
    cursor.execute('SELECT * FROM requests WHERE status = ?', ('Pending',))
    rows = cursor.fetchall()

# Writing data (auto-commits when with block exits)
with get_connection() as conn:
    conn.execute(
        'INSERT INTO requests (requester, category, amount) VALUES (?, ?, ?)',
        ('Taylor', 'Travel', 1200.0)
    )
⚠
Never use f-strings in SQL — state this clearly
Say: “Never build a SQL string by concatenating or f-string-ing user input. This is a SQL injection vulnerability. Always use ? placeholders and pass values as a tuple. This is non-negotiable in professional code and in this course.”
Section 2 — Build database.py Together

Build database.py Together 15–40 min

Live coding · All five required functions

🗣 Say before starting
“All five of these functions are required in your capstone. We are building them together now so you understand the pattern. Then you adapt them to your own schema.”
The five required functions — build together
import sqlite3

DB_FILE = 'data/requests.db'

def create_table():
    with sqlite3.connect(DB_FILE) as conn:
        conn.execute('''
            CREATE TABLE IF NOT EXISTS requests (
                id        INTEGER PRIMARY KEY AUTOINCREMENT,
                requester TEXT    NOT NULL,
                category  TEXT    NOT NULL,
                amount    REAL,
                status    TEXT    DEFAULT 'Pending',
                notes     TEXT
            )
        ''')

def add_record(requester, category, amount, notes=''):
    with sqlite3.connect(DB_FILE) as conn:
        conn.execute(
            'INSERT INTO requests (requester,category,amount,notes) VALUES(?,?,?,?)',
            (requester, category, amount, notes)
        )

def get_all_records():
    with sqlite3.connect(DB_FILE) as conn:
        conn.row_factory = sqlite3.Row
        cur = conn.execute('SELECT * FROM requests ORDER BY id DESC')
        return [dict(row) for row in cur.fetchall()]

def update_status(record_id, new_status):
    with sqlite3.connect(DB_FILE) as conn:
        conn.execute('UPDATE requests SET status=? WHERE id=?', (new_status, record_id))

def get_status_report():
    with sqlite3.connect(DB_FILE) as conn:
        cur = conn.execute(
            'SELECT status, COUNT(*) as count, SUM(amount) as total FROM requests GROUP BY status'
        )
        return cur.fetchall()
Section 3 — row_factory and Dict Conversion
3

row_factory and Dict Conversion 40–55 min

Explain why — Streamlit needs dicts

🗣 Say — why row_factory
“By default fetchall() returns tuples: (1, 'Taylor', 'Travel', 1200.0, 'Pending', ''). You access fields by index: row[1] is Taylor. If you reorder your columns, all your index references break. Setting row_factory to sqlite3.Row and wrapping in dict() gives you named access: record['requester']. Much more readable and robust.”
💡 Streamlit needs dicts
st.dataframe() accepts a list of dicts directly. If you return tuples from your database functions you will need to convert them in app.py. Return dicts from the database layer and app.py stays clean.
Section 4 — Testing Database Functions with tmp_path
4

Testing Database Functions with tmp_path 55–70 min

Show the pytest fixture pattern

🗣 Say — never test against the development database
“If your pytest tests INSERT records into your real requests.db, your development database fills up with test garbage. Use pytest's tmp_path fixture — it creates a fresh temporary folder for each test, isolated from everything else.”
Test pattern with tmp_path — show this pattern
def test_add_and_retrieve(tmp_path):
    db = tmp_path / 'test.db'
    create_table(db)
    add_record('Taylor', 'Travel', 1200, db_file=db)
    records = get_all_records(db_file=db)
    assert len(records) == 1
    assert records[0]['requester'] == 'Taylor'
⚠
Requires db_file parameter
For this pattern to work, every database function must accept an optional db_file parameter that defaults to DB_FILE. Show students how to add this parameter to each function. This is the design pattern that makes database code testable.
