Requirement-by-Requirement Walkthrough
1
All five database functions with docstrings and db_file parameter

Open database.py. Audit all five functions by name and structure.

🔍 What the grader checks
create_table(db_file=DB_FILE): uses CREATE TABLE IF NOT EXISTS
add_record(..., db_file=DB_FILE): inserts one record, all required fields as parameters
get_all_records(db_file=DB_FILE): returns list of dicts
update_status(record_id, new_status, db_file=DB_FILE): WHERE id = ?
get_status_report(db_file=DB_FILE): GROUP BY status with COUNT and SUM
All five have docstrings
All five accept an optional db_file parameter defaulting to DB_FILE
⚠ Common mistakes at this step
Missing db_file parameter on any function — tests cannot use tmp_path; partial credit (15/25)
create_table uses CREATE TABLE (not IF NOT EXISTS) — will error on second run
add_record hardcodes the table name or fields — not flexible for the student's schema
get_status_report returns all records instead of aggregates — check for GROUP BY in the SQL
2
No f-strings or concatenation in SQL

Search the entire database.py file for any f-string inside a SQL statement. This is a zero-tolerance requirement.

🔍 What the grader checks
Every SQL string uses ? placeholders: 'INSERT INTO ... VALUES (?, ?)'
All values passed as a tuple in the second argument to conn.execute()
No f-string with user data embedded in a SQL string
No string concatenation with + inside SQL strings
💡 How to check quickly
Search the file for f"... (f followed by quote) inside any execute() call. Also search for + inside execute() arguments. Either is a violation.
✅ Correct (full credit)
conn.execute('INSERT INTO requests (requester, amount) VALUES (?, ?)', (requester, amount))
❌ SQL injection (zero credit)
conn.execute(f"INSERT INTO requests (requester, amount) VALUES ({requester}, {amount})")
⚠ Common mistakes at this step
Even one f-string in SQL earns 0/20 for this requirement — no partial credit
Student uses %s formatting (old-style string format in SQL) — also a violation
String concatenation: 'WHERE status = ' + status — also a violation
3
get_all_records returns a list of dicts

Verify the function uses row_factory and dict() conversion.

🔍 What the grader checks
conn.row_factory = sqlite3.Row set before execute()
Return value is [dict(row) for row in cur.fetchall()]
Function returns a list (not a single dict, not a list of tuples)
Records ordered meaningfully (ORDER BY id DESC or similar)
✅ Full credit
conn.row_factory = sqlite3.Row cur = conn.execute('SELECT * FROM ... ORDER BY id DESC') return [dict(row) for row in cur.fetchall()]
❌ Returns tuples (no credit)
cur = conn.execute('SELECT * FROM ...') return cur.fetchall() # list of tuples, not dicts
⚠ Common mistakes at this step
row_factory set but dict() not used — Row objects returned, not plain dicts
fetchone() used instead of fetchall() — returns only one record
Returns None instead of empty list when table is empty
4
Five pytest tests using tmp_path

Open tests/test_database.py. Every test must use the tmp_path fixture and pass a fresh database path to every function call.

🔍 What the grader checks
5+ test functions, each accepting tmp_path as a parameter
Each test creates: db = tmp_path / 'test.db', then passes db as db_file
Tests cover: add+retrieve, default status, update_status, multiple records, status report grouping
No test uses the real DB_FILE — all use tmp_path
All tests pass
✅ Correct tmp_path usage
def test_add_and_retrieve(tmp_path): db = tmp_path / 'test.db' create_table(db) add_record('Taylor', 'Travel', 1200, db_file=db) records = get_all_records(db_file=db) assert len(records) == 1
❌ Missing tmp_path (no credit)
def test_add_record(): create_table() # uses real DB — pollutes development database add_record('Taylor', 'Travel', 1200)
⚠ Common mistakes at this step
Tests use the real database (no tmp_path) — 0/25 for this requirement
Only 3-4 tests — partial credit (15/25 for 4 tests, 10/25 for 3 tests)
Tests do not cover all 5 required types — partial credit proportional to coverage
5
test_script.py demonstrating all five functions end-to-end

Open test_script.py. Run it mentally — does it call all five functions in a logical sequence?

🔍 What the grader checks
create_table() called first
add_record() called at least twice with different data
get_all_records() called and results printed
update_status() called on one record
get_status_report() called and output printed
Output is readable and demonstrates real data flowing through the system
Known Common Mistakes
⚠ Known common mistakes (from the assignment page)
Using the development database in tests — always use tmp_path / 'test.db' in every test function
Returning tuples from get_all_records — must use row_factory = sqlite3.Row and dict(row)
update_status that updates every record instead of one — the WHERE clause must filter by id = ?
get_status_report that returns all records instead of grouped aggregates — must use GROUP BY status

Rubric discussion
The SQL injection requirement (20 pts) is zero-tolerance — any f-string in SQL earns 0/20. This is the most important security habit in the course and must be enforced strictly. The tmp_path requirement (25 pts) is also strictly graded — tests that use the real database are not acceptable because they couple tests to persistent state. However, if only 1-2 tests out of 5 use the real database, consider awarding 15/25.
