How to use this walkthrough
Week 14 is the most technically specific week of the capstone build. This walkthrough covers the sqlite3 connection pattern, SQL injection prevention, row_factory, and the tmp_path testing pattern. Q2 (SQL injection via f-strings) and Q3 (row_factory enabling named access) are the two concepts that most distinguish professional database code from beginner database code. Every essay question has a model answer that can be used directly as a grading key.
Question-by-question walkthrough
1
Multiple Choice — Auto-graded
Why should every database function use a "with sqlite3.connect(db_file) as conn:" context manager?
A.It is faster○
⚠ Context managers do not inherently make code faster — they ensure cleanup happens correctly.
B.It automatically commits on success and closes the connection when the block exits✓
C.It encrypts the database○
⚠ sqlite3 does not encrypt databases by default — encryption requires SQLCipher or similar.
D.It is required by Python 3○
⚠ Context manager syntax has been available since Python 2.6 — it is not a Python 3 requirement.
💬 Discussion — why B is correct

The context manager handles commit and close automatically. Without it you must manually call conn.commit() and conn.close() — forgetting either causes bugs or connection leaks.

2
Multiple Choice — Auto-graded
What is wrong with using an f-string inside a SQL statement?
A.Nothing — f-strings are fine○
⚠ There is nothing correct about f-strings in SQL with user data. This option should be clearly false.
B.It is a SQL injection vulnerability — user input embedded directly into SQL can execute malicious commands✓
C.f-strings are not supported in sqlite3○
⚠ f-strings do not cause SyntaxErrors — they cause security vulnerabilities at runtime.
D.It causes a SyntaxError○
⚠ f-strings work fine in sqlite3 — the issue is security, not compatibility.
💬 Discussion — why B is correct

Never use f-strings or string concatenation to build SQL with user data. Always use ? placeholders: conn.execute("INSERT ... VALUES (?, ?)", (name, amount)).

3
Multiple Choice — Auto-graded
What does conn.row_factory = sqlite3.Row change about fetchall()?
A.It makes queries faster○
⚠ row_factory can improve performance marginally but that is not its purpose.
B.It makes each row accessible by column name instead of by index position✓
C.It returns only the first row○
⚠ row_factory does not limit to the first row — that would be fetchone().
D.It formats numbers as currency○
⚠ row_factory does not format numbers — that is done in the application layer.
💬 Discussion — why B is correct

Without row_factory, fetchall() returns plain tuples accessed by index (fragile). With sqlite3.Row, rows are accessed by column name. dict(row) converts to a plain Python dict.

4
Multiple Choice — Auto-graded
Why does every database function accept an optional db_file=DB_FILE parameter?
A.To make the code longer○
⚠ Longer code is not the goal — testability is.
B.So tests can pass a tmp_path database instead of using the development database✓
C.Because sqlite3 requires it○
⚠ sqlite3 does not require a db_file parameter — it is a design choice for testability.
D.To support multiple database engines○
⚠ The db_file parameter is used with sqlite3 only — this course does not cover other engines.
💬 Discussion — why B is correct

Tests must use isolated temporary databases. If db_file is hardcoded, every test pollutes the development database. The optional parameter lets tests pass a fresh temp path.

5
Multiple Choice — Auto-graded
What does CREATE TABLE IF NOT EXISTS do differently from CREATE TABLE?
A.They are identical○
⚠ CREATE TABLE and CREATE TABLE IF NOT EXISTS produce identical results when the table does not exist — the difference only matters when it already exists.
B.IF NOT EXISTS prevents an error if the table already exists — safe to call every app startup✓
C.IF NOT EXISTS deletes the existing table first○
⚠ IF NOT EXISTS does not delete existing tables — it simply does nothing if the table exists.
D.IF NOT EXISTS is faster○
⚠ IF NOT EXISTS has no measurable performance impact.
💬 Discussion — why B is correct

CREATE TABLE alone errors if the table already exists. IF NOT EXISTS is idempotent — creates if missing, does nothing if present. Safe to call at every app startup.

6
Multiple Choice — Auto-graded
What does the ? placeholder in sqlite3 do?
A.It is a wildcard that matches any column○
⚠ ? is not a wildcard for columns — * is used for that (SELECT *).
B.It is a positional parameter — sqlite3 safely substitutes the value from the tuple, preventing SQL injection✓
C.It selects all columns○
⚠ ? does not select all columns — that is *.
D.It means NULL○
⚠ ? does not mean NULL — NULL is represented as None in Python.
💬 Discussion — why B is correct

The ? placeholder is replaced by the value at the corresponding position in the second argument tuple. sqlite3 handles quoting and escaping, preventing SQL injection.

7
Multiple Choice — Auto-graded
What does get_status_report() return?
A.All records sorted by status○
⚠ All records sorted by status is SELECT * FROM requests ORDER BY status — not a report.
B.One row per unique status with count and total amount✓
C.The most recent record○
⚠ The most recent record is SELECT * FROM requests ORDER BY id DESC LIMIT 1.
D.A list of status strings○
⚠ A list of status strings would be SELECT DISTINCT status FROM requests.
💬 Discussion — why B is correct

get_status_report() runs: SELECT status, COUNT(*), SUM(amount) FROM requests GROUP BY status. This returns one row per status with aggregate values.

8
Multiple Choice — Auto-graded
A student's get_all_records() returns a list of tuples. What is wrong and how is it fixed?
A.Nothing — tuples are correct○
⚠ Tuples are incorrect — they force index-based access which breaks when columns are reordered.
B.row_factory was not set. Fix: add conn.row_factory = sqlite3.Row before execute, then return [dict(row) for row in cursor.fetchall()]✓
C.The SELECT query is wrong○
⚠ The SELECT query is fine — the issue is the missing row_factory.
D.The db_file parameter is missing○
⚠ The db_file parameter is about testability — it would not prevent the tuples issue.
💬 Discussion — why B is correct

Without row_factory, sqlite3 returns tuples. Set row_factory = sqlite3.Row before executing, then convert with dict().

Code Trace / Short Answer — Manual grading
What is wrong with this function?
def add_record(name, amount): with sqlite3.connect(DB_FILE) as conn: conn.execute(f"INSERT INTO records VALUES ({name}, {amount})")
✓ Model Answer
It uses an f-string to build the SQL, which is a SQL injection vulnerability. Fix: use ? placeholders and pass values as a tuple.
