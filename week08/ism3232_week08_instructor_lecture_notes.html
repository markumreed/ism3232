Section 1 — Reading Tracebacks
1

Reading Tracebacks 0–15 min

Live demo · Produce an actual error · Read it together

🗣 Say — framing
“A traceback is not failure — it is information. Python is telling you the file, the line, and exactly what went wrong. Students who read tracebacks carefully fix bugs in minutes. Students who ignore them and immediately ask AI spend hours on the same bugs.”
✏ Write on board — bottom-up reading order
Read bottom-up: (1) error type and message → (2) file and line number → (3) what called it
🗣 Say — produce a live error
Write a short script with a deliberate KeyError (typo in a dictionary key). Run it. Show the traceback. Walk through it bottom-up out loud before fixing it.
💡 The three questions to ask when you see a traceback
1. What type of error is it? 2. Which file and line? 3. What was the call chain? Answer these three before touching anything.
Section 2 — Debugging with print()
2

Debugging with print() 15–35 min

Live demo · Show the workflow, not just the commands

🗣 Say — what debugging is
“Debugging is finding the gap between what you think the program contains and what it actually contains. print() lets you inspect values at specific execution points. Start at the error line and work backwards, printing variables until you find where the data became wrong.”
Strategic print debugging — demo live
def process_requests(requests):
    total = 0
    for req in requests:
        print(f'DEBUG: req = {req}')               # inspect each record
        amount = req['amount']
        print(f'DEBUG: amount = {amount} ({type(amount).__name__})')
        total += amount
    print(f'DEBUG: total = {total}')
    return total
# After fixing: remove all DEBUG lines before ruff + commit
⚠
Remove debug prints before submitting
Show ruff check catching a leftover debug print. Say: “Ruff will flag unused variables and some patterns. Leaving debug prints in submitted code is a workflow violation.”
Section 3 — Required AI Literacy Activity — Debug First, Then Ask
3

Required AI Literacy Activity — Debug First, Then Ask 35–55 min

Graded activity · Emphasise process not just the fix

⚠
Graded on process, not just the fixed code
Students who submit only the corrected file with no reflection receive no credit for the activity portion. Walk through the six steps explicitly.
🗣 Say — the point of this activity
“This activity teaches you to use AI as a tool for understanding, not as a replacement for thinking. The sequence matters: attempt first, document what you tried, then ask AI to explain the error — not to fix the code.”
✏ Write the six steps on board
1. Read the traceback. Write what you think is wrong.   2. Try to fix it yourself with print().   3. If stuck: ask AI to explain the error message ONLY.   4. Fix the code yourself.   5. Run pytest.   6. Write the reflection.
⚠
The prohibited prompt — say this out loud
“Fix this code for me.” This is not permitted. Show students the difference: “Explain this Python error without giving me the corrected code” is permitted. The reflection must document which prompt they used.
